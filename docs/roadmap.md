| 14 | Investigate replica-journal divergence at segment boundaries | Medium | Running `scripts/journal-verify.sh` against any tcp-dual-repl bench shows `error at entry 1: sequence duplicate: <N> already seen (immediately after seq <N>)` on **both** replicas, while the primary's journal verifies cleanly. Investigation needs: dump entries 1 and 2 of a replica live segment, compare to the corresponding bytes on primary, trace the receiver's per-event write path (`crates/server/src/replication/tcp_receiver.rs`) across a rotation. Correctness gap if real: a verified-bad replica could be promoted and start serving from a journal that doesn't match what primary acked. Closes one of the standing "replicas verify clean" claims in the README. |

## Rumcast Throughput ‚Äî Path to Kernel-TCP Parity

LAN bench results (busy-spin default, 16-client √ó 128-window throughput) show kernel-UDP rumcast at ~53k orders/s vs kernel-TCP (io_uring + 4 reader threads) at ~3.57M orders/s ‚Äî a ~67√ó throughput gap and ~1.5√ó single-message p50 gap. The single-message gap is mostly the cost of the rumcast reliability layer (envelope + framing + flow control) and is largely structural; the throughput gap is architectural and addressable. These items, in order, close it. DPDK is sequenced last on purpose ‚Äî without a known kernel-UDP per-core ceiling, DPDK numbers conflate two unknowns and become hard to defend in a sales conversation.

| # | Feature | Commercial value | Complexity | Value/effort | Why |
|---|---------|:---:|:---:|:---:|-----|
| 1 | SO_REUSEPORT session sharding (server) | High | Medium | ‚òÖ‚òÖ‚òÖ‚òÖ‚òÜ | Today the server runs one session-translator thread that does recv ‚Üí demux ‚Üí app ‚Üí mux ‚Üí send for *all* clients ‚Äî a single-core ceiling regardless of how many cores the box has. Bind N sockets to the same UDP port with `SO_REUSEPORT`; the kernel hashes flows across them. Run N session-translator shards (one per socket, pinned to its own core), each owning a disjoint set of sessions. This is the kernel-UDP equivalent of "scale out across cores" and a prerequisite for any honest comparison with multi-threaded TCP. Real customers run kernel UDP (Tier-2 commercial strategy), so this has direct revenue value. |
| 2 | Multi-threaded rumcast bench | Medium | Low | ‚òÖ‚òÖ‚òÖ‚òÜ‚òÜ | The bench is a single hot loop driving 16 sessions on one thread. Once #1 lands, the bench cannot saturate a sharded server without itself being multi-threaded. Drive disjoint session subsets from N bench threads with their own UDP sockets, aggregate latency histograms at the end. Without this, the server-side win in #1 is unmeasurable. |

## Deferred

Features targeting regulated venues, gateway responsibilities, or with limited near-term value. Will revisit when the core product is mature or a specific buyer requires them.

| Feature | Why deferred |
|---------|-------------|
| SPDK journal | Userspace NVMe driver for journal writes. Bypasses kernel block layer entirely. TCP pipelining already hides fsync latency (fsync and no-persist throughput converged at ~8M/s), so the kernel block layer is no longer a bottleneck. High complexity, minimal expected gain. |
| Adaptive overlapped io_uring journal writes | The non-aggressive form is **already shipped** in `run_uring`: the journal stage submits `WRITE+RWF_DSYNC` async via io_uring and continues encoding the next batch while the NVMe write is in flight. CQEs are reaped non-blocking at two points per loop iteration; backpressure only kicks in when the next batch fills before the previous fsync completes. What is still deferred is the *aggressive* double-buffer variant (always keep two writes inflight, even for tiny batches). That was built and reverted because events caught in a small overlapped batch had their cursor delayed by one extra NVMe latency, hurting p99.9. The next attempt should only overlap when `pending > 16` events. Branch preserved. |
| WRITE_FIXED for journal | Register batch buffers via `IORING_REGISTER_BUFFERS`, use `IORING_OP_WRITE_FIXED`. Skips `get_user_pages()` per SQE (~100-200ns). Failed for socket I/O on kernel 6.8 but works for file writes. |
| Vectored journal writes | Encode each event into its own stack buffer, pass as iovec array to `pwritev2`. Eliminates memcpy-per-event into the batch buffer. Small gain (~1-2¬µs/batch). |
| Verbatim byte-path journaling | Write the primary's encoded journal bytes directly on both primary and replica, skipping the decode/re-encode step on the replica side and an encode-from-struct pass on the primary. Saves CPU and reduces journal-stage work on the replay-critical path.
