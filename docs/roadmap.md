# Roadmap

Planned features sorted by value/complexity ratio for commercial readiness (exchange operators and investors).

## Active

Sequenced by dependency and priority — items higher up are unblocked
and ship first, items further down build on what came before.

| # | Feature | Complexity | Why |
|---|---------|:---:|-----|
| 1 | Publish DPDK numbers alongside kernel-TCP | Low | The README and current sales numbers are kernel-TCP only. Tier-1 TradFi venues all run on kernel bypass or specialized NICs — without DPDK figures the comparison is structurally incomplete and a buyer will discount the numbers accordingly. The DPDK transport already exists (server `dpdk` feature, `crates/bench/src/dpdk.rs`, exercised by `scripts/lan-bench-suite.sh`); this is harvesting numbers under the existing realistic flow generator and publishing them next to the kernel-TCP table in the README and `docs/benchmarking.md`. Pairs with #2 — without per-stage decomposition the DPDK win is reported as a black-box delta rather than attributable to NIC bypass. Independent of all engine work below. |
| 2 | Tick-to-trade decomposition in bench output | Medium | Today the bench reports full client-to-client round-trip latency. Exchange RFPs typically ask for the latency profile decomposed into NIC ingress → parse/decode → match → encode/respond → NIC egress, separately from the durability gate (journal + replication). The plumbing is partly there — `InputSlot::timestamp_ns` is captured at receipt and pipeline-stats tracks per-stage occupancy in aggregate — but per-request stage timestamps need to flow through the response path and into the bench histograms. Lets us report "gateway-to-match" the way CME / Nasdaq do, makes the kernel-TCP-vs-DPDK comparison from #1 attributable, and answers the most common diligence question without hand-waving. |
| 3 | Journal rotation soak + integration tests | Low | Runtime rotation now fires from a size threshold or operator `ROTATE` admin command, with multi-segment recovery walking archives and the live segment in order. Unit-level coverage is in place; what's missing is a long-run integration test that drives sustained traffic through one or more rotations on both primary and replica, verifies post-recovery state matches a no-rotation baseline, and exercises crash points at each rotation phase (size trigger, rename, new-segment open, snapshot mid-write). Production-readiness gate for any 24/7 deployment. |
| 4 | Configurable durability ack policy | Medium | The response stage today picks between two hardcoded gates (`--quorum-durability` flips between `min(journal, repl_min)` and `max(repl_min, min(journal, repl_max))`), and `--async-replica-ack` further bends the rules per-node. Generalise into a structured policy expressed against per-node durability levels — `in_memory` (event accepted by the pipeline), `journaled` (queued for write), `fsynced` (PLP-confirmed) — and aggregate predicates (e.g. *`ack when ≥1 node fsynced AND ≥1 other node in_memory`*, or *`ack when ≥2 nodes fsynced`*). Operators tune the durability/latency tradeoff per their risk profile and PLP topology without code changes; subsumes both the existing `--quorum-durability` toggle and the async-ack flags (including the previously-roadmapped async-primary-ack — that's just `ack when primary ≥ in_memory AND ≥1 replica ≥ in_memory` in this framework). Needs each replica to publish a per-level cursor (already partly done — `replication_cursor` is `in_memory`-ish, journal cursor is `fsynced`) and a small policy evaluator on the response gate. |
| 5 | Replicas pull configuration from primary | Medium | Today each node reads its own `ServerConfig` (instruments, fee schedules, auth keys, etc.) from local files; operators must keep the primary's and every replica's configs identical or replication diverges silently. Have the replica fetch the canonical config from the primary as part of the handshake, after auth but before journal catch-up — the replica only owns transport-level config (listen addr, replication addr, certs). Eliminates a silent-divergence class of incident, makes adding a replica a one-step operation, and is a prerequisite for any meaningful Raft integration (#7) where the cluster owns shared state. |
| 6 | Raft cluster integration | Very high | Replace the bespoke 1+1 / 1+2 replication with Raft consensus over the journal. Gives well-known durability guarantees (regulators recognise it), automatic leader election, formal split-brain prevention, and unlocks 3+ replica topologies — currently structurally impossible. Major undertaking: needs a Raft library (or in-house implementation), integration with the event-sourced pipeline so log entries map onto journal events, and a migration story for existing primary+replica deployments. Prerequisite for the auto-promotion item below. |
| 7 | Auto-promotion via Raft leader election | Medium | Once #6 (Raft) lands, replace the operator-driven `PROMOTE` admin endpoint with Raft's built-in leader-election semantics: when the current leader becomes unreachable, the surviving quorum elects a new one and the order-entry / replication addresses follow leadership automatically. Eliminates the human-in-the-loop window during failover (currently several seconds, dominated by operator reaction time) and the silent-data-loss class the manual flow guards against. Needs: leadership-aware bind/unbind on the order-entry transports, client-side reconnect to follow the new leader (the bench client and FIX gateway already reconnect on transport errors — re-resolving the leader is the new piece), and a fencing mechanism so a partitioned ex-leader stops acking before its lease expires. |
| 8 | Replicas refuse gateway connections with a primary-address redirect | Low | When a client (FIX gateway, bench, native rumcast/TCP client) connects to a node that isn't currently the leader, the node should reject the handshake with a structured redirect carrying the current leader's order-entry address rather than silently accepting and then dropping orders, or rejecting with an opaque error. Add a `Redirect { primary_addr }` response variant and emit it from the auth path on any non-leader node; clients reconnect to the address in the frame. Closes the "ops accidentally pointed me at a replica" footgun on its own, and pairs with #7 to give clients a clean follow-the-leader story across failover (replicas know the new leader as soon as Raft elects it; the next gateway reconnect lands in the right place automatically). Independent of #6/#7 — works today using static knowledge of the configured primary. |
| 9 | Brand setup (domain, GitHub org, email) | Low | Register melin.io/melin.com, set up contact@ email, create GitHub org, transfer repo, switch commit email going forward. Do not rewrite history. Independent of the engine work above; sequenced last because it doesn't unblock anything technical. |

## Rumcast Hardening

Follow-ups to take the rumcast transport (Ed25519 + X25519 auth, per-session BLAKE3 keyed-MAC envelopes, multi-client demux) from feature-complete to production-ready. The foundation is on `main`; these items make it deployable for real exchange operators.

| # | Feature | Commercial value | Complexity | Value/effort | Why |
|---|---------|:---:|:---:|:---:|-----|
| 1 | Per-deployment tunables (`MAX_SESSIONS`, `IDLE_TIMEOUT`, `TERM_LENGTH`) | Medium | Low | ★★★★☆ | All three are hardcoded `const`s in `rumcast_transport.rs` (with `TODO(config)` markers from Phase 3). Operators with small boxes need lower `MAX_SESSIONS`; trading deployments with clients quietly tracking market state during off-hours need longer `IDLE_TIMEOUT`; fully-saturating 10GbE workloads need larger `TERM_LENGTH` (current 1 MiB is borderline at 1.25 MiB BDP). Mechanical refactor: promote each to `ServerConfig`, plumb through, document defaults. |
| 2 | Per-session observability via health endpoint | Medium | Low | ★★★★☆ | The rumcast `Counters` struct already aggregates per-tick stats (fragments accepted, NAKs sent, sessions created/rejected, gaps detected, etc.) and the muxers wire it through, but the existing health endpoint doesn't expose them. Operators debugging a multi-client deployment can't see per-session NAK rates, idle eviction counts, or handshake failures without reading server logs. Surface via the existing Prometheus-text health endpoint. |
| 3 | Token rotation / TTL | Medium | Medium | ★★★☆☆ | Each authenticated rumcast session derives a BLAKE3 keyed-MAC token at handshake time and uses it forever. A long-running session keeps the same key indefinitely; if the token is ever leaked (memory dump, side-channel) an attacker can forge messages until the session ends. Add server-initiated periodic re-handshake (every N min): server emits a new control message, client responds with a fresh `ChallengeResponse`, both sides switch to the new token at a fixed sequence boundary. No wire-format change to Data frames. Defense-in-depth — protects long-lived connections. |
| 4 | Async / yielding catch-up on the rumcast sender | Medium | Medium | ★★★☆☆ | `replication/rumcast_sender.rs::run_catchup_or_snapshot` runs synchronously inside the single-thread tick loop. A fresh replica catching up from a 10 GiB journal is ~160k publish iterations all blocking the same thread that drains the live replica's NAKs/heartbeats. Acceptable when the other slot is empty; potentially bad when one replica is in steady-state and another joins. Fix: yield muxer ticks every N publish iterations (smaller change), or run catch-up on a per-slot worker thread with a bounded queue. |
| 5 | Promotion handoff integration test | High | Low | ★★★★☆ | `run_rumcast_replica` → `run_rumcast_primary_with_state` is wired but untested. Same for the post-`catch_unwind` promote listener path in `promote.rs`. An integration test that triggers `promote_flag` and verifies the replica binds the order-entry port + accepts a trader connection would catch regressions on a critical-path failover transition. |
| 6 | Replication-over-rumcast test breadth | Medium | Medium | ★★★☆☆ | `crates/server/tests/rumcast_replication_smoke.rs` only verifies handshake + genesis byte-equality. Untested: live data flow (orders → primary → replica DataBatch → ack chain), snapshot transfer end-to-end, replica reconnect with backoff, concurrent 2-replica setup, ring backpressure eviction, and the auth-side `Challenged` (5s) / `Authenticated` (300s) idle-timeout sweeps. The bench suite covers data flow on the LAN, but per-scenario regression tests inside `cargo test` would catch wire-format breaks earlier. |
| 7 | `SnapshotUnavailable` error frame | Low | Low | ★★★☆☆ | When the primary's `run_catchup_or_snapshot` finds `can_catch_up=false` AND no snapshot file exists, it returns false silently. The replica eventually times out + reconnects in a loop, only seeing the cause in primary-side logs. Add a `SnapshotUnavailable` `PrimaryMessage` variant so the replica logs the reason and operators can diagnose without SSH'ing into the primary. |
| 8 | `AuthFailed` integration test | Low | Low | ★★★☆☆ | `protocol/src/session.rs` has 23 unit tests for the envelope path and `crates/protocol/src/fuzz_tests.rs` is fuzzed, but there's no end-to-end test for "server rejects bad signature → client receives `AuthFailed` → connection closes cleanly". Wires the sad path. |
| 9 | Bench-suite restore-step failure handling | Low | Low | ★★★★☆ | `scripts/lan-bench-suite.sh` line ~514 swallows failures from the post-rumcast default-features rebuild via `wait "$pid" \|\| true`. If the restore fails after the rumcast build, downstream `tcp` transports in the same matrix will silently run with a rumcast-only binary. Add an explicit failure path with logging and a non-zero exit. Five-line shell fix; high value/effort because the silent-failure mode is genuinely confusing. |
| 10 | Rumcast crate stress + edge-case tests | Low | Medium | ★★☆☆☆ | The `melin-rumcast` crate has solid coverage of the happy path (wire round-trip, term rotation, NAK retransmit, flow control, multi-session demux), but three corners are untested: multicast fan-out with multi-subscriber NAK suppression + jitter convergence, term-wraparound stress (`term_id` wrap at `u32::MAX`), and the `published_fragment` HWM-based gap detection's documented "HWM doesn't decrease on interior fill" simplification. Add stress tests; add a roadmap-style code comment documenting that proper Aeron-style per-range loss detection is deferred. |
| 11 | Pool reassembly buffers in `rumcast_receiver::poll_reassembled` | Low | Low | ★★★☆☆ | Each rumcast data frame in `streaming_loop`, `wait_for_message`, and the stash path currently allocates a `Vec<u8>` (`payload.to_vec()` / `inner.to_vec()`). Fine for catch-up bursts, but on the live-streaming hot path every frame allocates. Replace with a pooled or refcounted-slice (`Bytes`-style) buffer to drop per-frame allocations to zero in steady state. Correctness is unaffected — pure throughput/latency optimization. |
| 12 | Couple rumcast keepalive interval and `LIVE_ACK_TIMEOUT` via shared constant | Low | Low | ★★★★☆ | `rumcast_receiver.rs::streaming_loop` sends keepalive acks every 1s; `rumcast_sender.rs::LIVE_ACK_TIMEOUT` evicts a slot after 3s without an ack. The 3× margin is correct but the two constants live in different files with no comment linking them — silent drift could evict healthy replicas. Promote to a shared constant in `replication/mod.rs` (or a doc-comment cross-reference) so the relationship is explicit. |
| 13 | MuxedSender active-sessions tracking | Medium | Medium | ★★★☆☆ | `MuxedSender::tick` walks every session's `drain_data` + `maybe_send_periodic` every tick, regardless of whether the session has new bytes to emit. At ~4 replicas this is invisible; at 10k+ authenticated OE-gateway sessions the per-tick HashMap walk burns L1 / branch-predictor budget on idle sessions. Maintain an `active_sessions: VecDeque<u32>` of session_ids that have published since the last drain, repopulated by `try_claim` succeeding via a back-channel SPSC or a per-session "dirty" bit. Heartbeats stay on a separate slow timer so idle sessions still keep their liveness signal. Scale-dependent — only matters once OE-gateway concurrent session counts cross ~1k. |
| 14 | Replication-sender thread split | Medium | High | ★★★☆☆ | `rumcast_sender.rs::run_sender_rumcast` runs six unrelated state machines on one thread (muxer ticks, new-session detection, inbound auth, per-slot live streaming, heartbeats, sweeps). Fine at Tier-2 throughput; at sustained 1M+ ops/s this thread becomes the bottleneck even after the sendmmsg / recv_batch wins on `main`. Split into a poller (already lives in the io_uring endpoint), an auth/control thread, and a per-slot replication-stream thread, with SPSC rings between stages. Same shape as the engine pipeline, just applied to replication. Defer until profiling shows the single-thread limit is the binding constraint. |

## Rumcast Throughput — Path to Kernel-TCP Parity

LAN bench results (busy-spin default, 16-client × 128-window throughput) show kernel-UDP rumcast at ~53k orders/s vs kernel-TCP (io_uring + 4 reader threads) at ~3.57M orders/s — a ~67× throughput gap and ~1.5× single-message p50 gap. The single-message gap is mostly the cost of the rumcast reliability layer (envelope + framing + flow control) and is largely structural; the throughput gap is architectural and addressable. These items, in order, close it. DPDK is sequenced last on purpose — without a known kernel-UDP per-core ceiling, DPDK numbers conflate two unknowns and become hard to defend in a sales conversation.

| # | Feature | Commercial value | Complexity | Value/effort | Why |
|---|---------|:---:|:---:|:---:|-----|
| 1 | SO_REUSEPORT session sharding (server) | High | Medium | ★★★★☆ | Today the server runs one session-translator thread that does recv → demux → app → mux → send for *all* clients — a single-core ceiling regardless of how many cores the box has. Bind N sockets to the same UDP port with `SO_REUSEPORT`; the kernel hashes flows across them. Run N session-translator shards (one per socket, pinned to its own core), each owning a disjoint set of sessions. This is the kernel-UDP equivalent of "scale out across cores" and a prerequisite for any honest comparison with multi-threaded TCP. Real customers run kernel UDP (Tier-2 commercial strategy), so this has direct revenue value. |
| 2 | Multi-threaded rumcast bench | Medium | Low | ★★★☆☆ | The bench is a single hot loop driving 16 sessions on one thread. Once #1 lands, the bench cannot saturate a sharded server without itself being multi-threaded. Drive disjoint session subsets from N bench threads with their own UDP sockets, aggregate latency histograms at the end. Without this, the server-side win in #1 is unmeasurable. |
| 3 | Wire io_uring into the rumcast hot path | Medium | Medium | ★★★☆☆ | An io_uring endpoint already exists in `crates/rumcast/src/io_uring_endpoint.rs` but isn't wired into the bench / server send/recv path the way the TCP transport's io_uring path is. Submission/completion + multi-shot recv + registered buffers eliminate per-packet `sendmmsg`/`recvmmsg` syscall cost and bring the rumcast path onto the same kernel mechanism the TCP path uses — making the kernel-UDP-vs-kernel-TCP comparison apples-to-apples. |
| 4 | DPDK kernel bypass for rumcast | High | Very high | ★★★☆☆ | Once #1–#3 ship and the kernel-UDP per-core ceiling is known, run rumcast over DPDK. Eliminates the kernel network stack entirely (no syscalls, no skb allocation, no NAPI scheduling) — leaves a clean measurement of "rumcast protocol overhead vs TCP protocol overhead" without kernel-stack noise on the UDP side. Doubles as Tier-1 acquisition optionality. Sequence after kernel-UDP optimizations so the win is attributable. |

## Deferred

Features targeting regulated venues, gateway responsibilities, or with limited near-term value. Will revisit when the core product is mature or a specific buyer requires them.

| Feature | Why deferred |
|---------|-------------|
| SPDK journal | Userspace NVMe driver for journal writes. Bypasses kernel block layer entirely. TCP pipelining already hides fsync latency (fsync and no-persist throughput converged at ~8M/s), so the kernel block layer is no longer a bottleneck. High complexity, minimal expected gain. |
| Adaptive overlapped io_uring journal writes | The non-aggressive form is **already shipped** in `run_uring`: the journal stage submits `WRITE+RWF_DSYNC` async via io_uring and continues encoding the next batch while the NVMe write is in flight. CQEs are reaped non-blocking at two points per loop iteration; backpressure only kicks in when the next batch fills before the previous fsync completes. What is still deferred is the *aggressive* double-buffer variant (always keep two writes inflight, even for tiny batches). That was built and reverted because events caught in a small overlapped batch had their cursor delayed by one extra NVMe latency, hurting p99.9. The next attempt should only overlap when `pending > 16` events. Branch preserved. |
| WRITE_FIXED for journal | Register batch buffers via `IORING_REGISTER_BUFFERS`, use `IORING_OP_WRITE_FIXED`. Skips `get_user_pages()` per SQE (~100-200ns). Failed for socket I/O on kernel 6.8 but works for file writes. |
| Vectored journal writes | Encode each event into its own stack buffer, pass as iovec array to `pwritev2`. Eliminates memcpy-per-event into the batch buffer. Small gain (~1-2µs/batch). |
| Verbatim byte-path journaling | Write the primary's encoded journal bytes directly on both primary and replica, skipping the decode/re-encode step on the replica side and an encode-from-struct pass on the primary. Saves CPU and reduces journal-stage work on the replay-critical path. Attempted once as a replica-only optimization (branch `replica-verbatim-journal`) but created an asymmetry that diverged fsync batching behavior between primary and replica — reverted in favor of a symmetric decode/re-encode on both sides. Revisit only with a plan to apply it on both sides simultaneously, preserving per-fsync size bounds. |
| io_uring RECVSEND_FIXED_BUF | `IORING_RECVSEND_FIXED_BUF` for registered buffer recv/send. Returned EINVAL on kernel 6.8; should work on 6.10+. Also test `IORING_RECVSEND_BUNDLE` for batched recv. Est. 15-25% throughput. |
| Dual-NVMe journal hedging | Two journal threads on separate NVMe drives, response stage gates on the fastest. Cuts tail latency from P(slow) to P(slow)². Free durability redundancy. Low complexity but requires a second NVMe slot. Revisit when journal fsync is the dominant tail contributor. |
| AF_XDP transport | DaMoN '25 found AF_XDP disappoints vs DPDK for small-message request-response workloads. DPDK transport already in progress. Revisit if DPDK proves insufficient. |
| Per-account trading permissions | Gateway concern — each firm's gateway instance restricts which accounts that connection can trade. Multi-tenant access control. |
| Replica analytics (6 items) | External service — throughput counters, latency histograms, volume/book depth analytics, audit trail queries, fee/PnL. Consumes the journal stream, not engine code. |
| Output event log | Regulatory audit trail. Depends on output event channel. |
| Subscription management | Gateway concern — the engine broadcasts, the gateway filters per-subscriber. |
| Iceberg orders | Niche — only matters for venues with institutional flow. |
| Auction mechanisms | Regulated venues only. Massive complexity (state machine, indicative pricing, uncrossing). |
| Replication chain hash verification | The wire protocol carries chain hashes in handshakes, heartbeats, and data batches, but the replica never verifies them against its local journal state. The primary unconditionally sends `StreamStart` without checking the replica's reported hash. Implementing verification would detect silent replica divergence (bit-rot, misrouted journal files) and trigger automatic snapshot re-sync. Deferred because validating a hash at an arbitrary historical sequence requires either a sequence→hash index or a journal re-scan. |
| Failover detection + promotion | Leader election, split-brain prevention. Distributed systems hard mode — manual promotion covers the MVP. |
| Network partition handling | Fencing, quorum-based decisions. Same as above — extremely complex. |
| Chain replication | Chain replication (primary → replica A → replica B) reduces primary fan-out. Dual parallel replication (up to 2 replicas) is implemented; chain topology is deferred. |
| Position/exposure limits | Important for derivatives, less so for spot. Defer until a derivatives buyer needs it. |
| Tiered fee schedules | Volume-based tiers and per-account overrides. Can be implemented outside Melin — a fee service looks up the account's tier and sets the rate via the existing per-instrument fee API. |
| TLS | Most exchange deployments use VLAN instead. Only needed for compliance-driven buyers. |
| Hybrid UDP multicast + TCP recovery for event channel | Current event channel is pure TCP. Multicast would reduce latency for co-located subscribers but adds complexity (gap detection, retransmit). Defer until a buyer needs sub-microsecond market data. |
