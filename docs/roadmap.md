# Roadmap

Planned features sorted by value/complexity ratio for commercial readiness (exchange operators and investors).

## Active

Sequenced by dependency and priority — items higher up are unblocked
and ship first, items further down build on what came before.

| # | Feature | Complexity | Why |
|---|---------|:---:|-----|
| 1 | Root-cause the 1 Hz / ~500 ms p99.99 spike under O_DIRECT | Medium | Under `O_DIRECT` (the production-correct durability path on PLP NVMe), every tcp-dual-repl bench shows periodic p99.99 stalls of 500–770 ms at **~1.000 s** cadence; first spike at ~1.961 s into the bench. Reproduces primary-only (no replicas), at 8 K orders/sec (single workload) and at 2.5 M orders/sec (throughput workload) with the same magnitude — so the cause is timer-driven, not load-driven. **Eliminated:** rotation disabled (`--max-journal-mib 0`), segment preparer off, `prefault_pages` stripped, io-wq async fallback (bpftrace: 0 `io_uring_queue_async_work` events over 30 s of steady-state, with 513 K submits and 768 K completions), C-state exit (cpuidle `current_driver = none`), `vm.stat_interval` 1 Hz timer (bumped to 60 s, no change), `numa_balancing` (disabled, single NUMA node), NVMe APST / runtime PM tweaks (no change), per-CPU dio kworker / NVMe-queue-1 IRQ contention on the journal core (moved journal from core 1 → core 12; p99.99 unchanged at 765 ms — spike does not follow the journal thread). Boot config already eliminates the usual kernel-timer suspects: `isolcpus=1-23`, `nohz_full=1-23`, `rcu_nocbs=1-23`, `nowatchdog`, `processor.max_cstate=1`, `cpufreq.default_governor=performance`, `transparent_hugepage=never`, `skew_tick=1`. ext4 mount is already tuned: `journal_async_commit,nobarrier,commit=300,data=writeback` with extents pre-zeroed via `FALLOC_FL_ZERO_RANGE`. **What we have:** `matching: disruptor wakeup` max = 134 ms (~14 % of the spike), so the rest lives in the journal or response stages. **What we do not have:** `latency-trace` only instruments matching today — journal-stage and response-stage start/finish hooks are missing. Next steps (both heavier than bench probes — pick one): (a) add `register_stage` + `record_elapsed` calls around the journal stage's submit→CQE window and around the response stage's gate→flush window (~50 LoC per stage, runnable against the existing bench harness), or (b) kernel-side ftrace `function_graph` over `ext4_*,iomap_*,nvme_*,blk_mq_*` during a spike window and look for a 500 ms-bracket. Commercial impact: until this is closed, the "real durable" performance numbers in the README carry a 500 ms p99.99 we cannot explain — every diligence call will ask about it. Pre-requisite for honest O_DIRECT figures; until then the published numbers run with `no-o-direct` (buffered, not actually durable on pwrite return). |
| 2 | Publish DPDK numbers alongside kernel-TCP | Low | The README and current sales numbers are kernel-TCP only. Tier-1 TradFi venues all run on kernel bypass or specialized NICs — without DPDK figures the comparison is structurally incomplete and a buyer will discount the numbers accordingly. The DPDK transport already exists (server `dpdk` feature, `crates/bench/src/dpdk.rs`, exercised by `scripts/lan-bench-suite.sh`); this is harvesting numbers under the existing realistic flow generator and publishing them next to the kernel-TCP table in the README and `docs/benchmarking.md`. Pairs with #3 — without per-stage decomposition the DPDK win is reported as a black-box delta rather than attributable to NIC bypass. Independent of all engine work below. |
| 3 | Tick-to-trade decomposition in bench output | Medium | Today the bench reports full client-to-client round-trip latency. Exchange RFPs typically ask for the latency profile decomposed into NIC ingress → parse/decode → match → encode/respond → NIC egress, separately from the durability gate (journal + replication). The plumbing is partly there — `InputSlot::timestamp_ns` is captured at receipt and pipeline-stats tracks per-stage occupancy in aggregate — but per-request stage timestamps need to flow through the response path and into the bench histograms. Lets us report "gateway-to-match" the way CME / Nasdaq do, makes the kernel-TCP-vs-DPDK comparison from #2 attributable, and answers the most common diligence question without hand-waving. |
| 4 | Journal rotation soak + integration tests | Low | Runtime rotation now fires from a size threshold or operator `ROTATE` admin command, with multi-segment recovery walking archives and the live segment in order. Unit-level coverage is in place; what's missing is a long-run integration test that drives sustained traffic through one or more rotations on both primary and replica, verifies post-recovery state matches a no-rotation baseline, and exercises crash points at each rotation phase (size trigger, rename, new-segment open, snapshot mid-write). Production-readiness gate for any 24/7 deployment. |
| 5 | ~~Configurable durability ack policy~~ (DONE) | Medium | Landed on `feat/durability-policy`. Operators set `--durability-policy <STRING>` to one or more `<level>>=<n>[ best_effort]` clauses joined with `&&` over per-node durability levels (`persisted` for PLP-confirmed NVMe writes, `in_memory` for pre-journal receipt). The optional `best_effort` keyword marks a clause as degrade-friendly, clamping to the connected cluster shape rather than failing closed — strictly stronger than the legacy auto-degrade in 1+2 deployments. The legacy `--quorum-durability` and `--async-replica-ack` flags were removed; their semantics are subsumed by `persisted>=2 best_effort` (default) and `in_memory>=N` respectively. Each replica's `Ack` frame now carries both an `acked_sequence` (persisted) and `in_memory_sequence` cursor, so the gate can evaluate any policy without separate ack streams. Active degradation surfaces on `/healthz` as `melin_durability_policy_degraded` and emits a periodic warn. Follow-up items in `docs/durability-policy-followups.md` cover idle-stage observability, ack-on-receive plumbing for the `in_memory>=N` fast path, and commercial polish (named modes, degrade floor). |
| 6 | ~~Ack-on-receive + three-tier durability menu~~ (DONE) | Medium | Landed on `feat/ack-on-receive`. Every receiver (TCP, DPDK) emits a dual-track `Ack { acked_sequence, in_memory_sequence }` so `in_memory>=N` policies produce the latency they promise. The operator surface is a single `--durability-mode <local\|hybrid\|durably-replicated>` flag, default `hybrid` (one durable copy + one in-memory ack from another node — single-failure-safe with a ~80 µs RAM-only window). `Local` is single-node durability for dev/test; `DurablyReplicated` requires two persisted copies. Strict fail-closed: the gate stalls when the configured mode is structurally unsatisfiable, and the matching stage halts new orders when all replicas disconnect, surfacing the halt reason to clients via a dedicated rejection-bypass carve-out on the durability gate (no silent TCP stalls). The operator can also swap modes at runtime via a signed admin `DURABILITY <mode>` command — same channel as `PROMOTE` / `ROTATE` — driving the standard failover playbook: promoted node resumes trading at reduced durability in seconds without restart, target mode restored once peers reattach. Each runtime swap is INFO-logged on the audit trail. Open follow-up in `docs/durability-policy-followups.md`: bench the actual ack rate under realistic load and decide on a 50–100 µs minimum-interval throttle. Bench numbers carry the documented ~50–80 µs regression vs published figures until that lands. |
| 7 | Replicas pull configuration from primary | Medium | Today each node reads its own `ServerConfig` (instruments, fee schedules, auth keys, etc.) from local files; operators must keep the primary's and every replica's configs identical or replication diverges silently. Have the replica fetch the canonical config from the primary as part of the handshake, after auth but before journal catch-up — the replica only owns transport-level config (listen addr, replication addr, certs). Eliminates a silent-divergence class of incident, makes adding a replica a one-step operation, and is a prerequisite for any meaningful Raft integration (#8) where the cluster owns shared state. |
| 8 | Raft cluster integration | Very high | Replace the bespoke 1+1 / 1+2 replication with Raft consensus over the journal. Gives well-known durability guarantees (regulators recognise it), automatic leader election, formal split-brain prevention, and unlocks 3+ replica topologies — currently structurally impossible. Major undertaking: needs a Raft library (or in-house implementation), integration with the event-sourced pipeline so log entries map onto journal events, and a migration story for existing primary+replica deployments. Prerequisite for the auto-promotion item below. |
| 9 | Auto-promotion via Raft leader election | Medium | Once #8 (Raft) lands, replace the operator-driven `PROMOTE` admin endpoint with Raft's built-in leader-election semantics: when the current leader becomes unreachable, the surviving quorum elects a new one and the order-entry / replication addresses follow leadership automatically. Eliminates the human-in-the-loop window during failover (currently several seconds, dominated by operator reaction time) and the silent-data-loss class the manual flow guards against. Needs: leadership-aware bind/unbind on the order-entry transports, client-side reconnect to follow the new leader (the bench client and FIX gateway already reconnect on transport errors — re-resolving the leader is the new piece), and a fencing mechanism so a partitioned ex-leader stops acking before its lease expires. |
| 10 | Replicas refuse gateway connections with a primary-address redirect | Low | When a client (FIX gateway, bench, native TCP client) connects to a node that isn't currently the leader, the node should reject the handshake with a structured redirect carrying the current leader's order-entry address rather than silently accepting and then dropping orders, or rejecting with an opaque error. Add a `Redirect { primary_addr }` response variant and emit it from the auth path on any non-leader node; clients reconnect to the address in the frame. Closes the "ops accidentally pointed me at a replica" footgun on its own, and pairs with #9 to give clients a clean follow-the-leader story across failover (replicas know the new leader as soon as Raft elects it; the next gateway reconnect lands in the right place automatically). Independent of #8/#9 — works today using static knowledge of the configured primary. |
| 11 | Brand setup (domain, GitHub org, email) | Low | Register melin.io/melin.com, set up contact@ email, create GitHub org, transfer repo, switch commit email going forward. Do not rewrite history. Independent of the engine work above; sequenced last because it doesn't unblock anything technical. |
| 12 | Provision benchmark hosts with PLP NVMe drives | Low | The current Cherry rig ships consumer Toshiba KXG60 NVMes with no Power Loss Protection. The journal writer is `O_DIRECT` mandatory (since `e1ab519`), so every batch flush pays full device-side latency on these drives — measured ~4× throughput drop and >100 ms tail spikes vs `no-persist` runs. The `no-o-direct` cargo feature lets us bench around it, but the published persist numbers will keep underrepresenting the engine until we run on enterprise drives (Kioxia CD6/CM6, Samsung PM9A3, Solidigm D7-PS1010, Micron 7400/9400). Source PLP-equipped hosts on Cherry/Hetzner/OVH and rerun the LAN bench suite — those become the headline persist figures for the README and `docs/benchmarking.md`. |
| 13 | Replica chain-hash validation at handshake | Medium | A replica connecting with a divergent journal (e.g. previously synced to a different primary, or with corrupted history) is currently accepted without warning — the primary unconditionally sends `StreamStart` and streams new events on top of stale state. The `Handshake` already carries the replica's `chain_hash` at its `last_sequence` and the `HashMismatch` frame exists in the wire protocol, but neither side acts on a mismatch. Correctness gap: post-failover, the promoted node holds a journal that doesn't match the events clients were told about. Two implementation shapes: **(a) tip-only check** — primary sends its current `chain_hash` + tip sequence in `StreamStart`, replica verifies when its `last_sequence == primary's tip`; catches the common reconnect case (~150 LOC, half a day, no journal-crate changes); **(b) full arbitrary-N check** — new journal API that scans from the nearest Checkpoint ≤ `last_sequence` forward, reproducing the writer's BLAKE3 accumulation rule bit-for-bit; catches divergent replicas that are also behind (~320 LOC, 1–2 days, journal-crate surface area). Recommend (a) first to close the high-frequency gap, (b) when a customer needs hardened bit-rot detection across long reconnect gaps. Independent of all engine work above. |
| 14 | Investigate replica-journal divergence at segment boundaries | Medium | Running `scripts/journal-verify.sh` against any tcp-dual-repl bench shows `error at entry 1: sequence duplicate: <N> already seen (immediately after seq <N>)` on **both** replicas, while the primary's journal verifies cleanly. The pattern (two entries at the same sequence, at offset 1 of the *live* segment — i.e. the most-recently-rotated file) reproduces identically on pre- and post-prestage builds, so it predates the rotation fast path and is not caused by it. The verify script reports `PARTIAL MATCH` and exits 0 when chain hashes match, which is why no prior bench surfaced it. Hypothesis: the replica receiver's writer re-emits a `GenesisHash` (or analogous transparent entry) at the boundary sequence on rotation, then writes the next replicated event at the same seq instead of `seq+1`. Investigation needs: dump entries 1 and 2 of a replica live segment, compare to the corresponding bytes on primary, trace the receiver's per-event write path (`crates/server/src/replication/tcp_receiver.rs`) across a rotation. Correctness gap if real: a verified-bad replica could be promoted and start serving from a journal that doesn't match what primary acked. Closes one of the standing "replicas verify clean" claims in the README. |

## Deferred

Features targeting regulated venues, gateway responsibilities, or with limited near-term value. Will revisit when the core product is mature or a specific buyer requires them.

| Feature | Why deferred |
|---------|-------------|
| SPDK journal | Userspace NVMe driver for journal writes. Bypasses kernel block layer entirely. TCP pipelining already hides fsync latency (fsync and no-persist throughput converged at ~8M/s), so the kernel block layer is no longer a bottleneck. High complexity, minimal expected gain. |
| Adaptive overlapped io_uring journal writes | The non-aggressive form is **already shipped** in `run_uring`: the journal stage submits `WRITE+RWF_DSYNC` async via io_uring and continues encoding the next batch while the NVMe write is in flight. CQEs are reaped non-blocking at two points per loop iteration; backpressure only kicks in when the next batch fills before the previous fsync completes. What is still deferred is the *aggressive* double-buffer variant (always keep two writes inflight, even for tiny batches). That was built and reverted because events caught in a small overlapped batch had their cursor delayed by one extra NVMe latency, hurting p99.9. The next attempt should only overlap when `pending > 16` events. Branch preserved. |
| WRITE_FIXED for journal | Register batch buffers via `IORING_REGISTER_BUFFERS`, use `IORING_OP_WRITE_FIXED`. Skips `get_user_pages()` per SQE (~100-200ns). Failed for socket I/O on kernel 6.8 but works for file writes. |
| Vectored journal writes | Encode each event into its own stack buffer, pass as iovec array to `pwritev2`. Eliminates memcpy-per-event into the batch buffer. Small gain (~1-2µs/batch). |
| Verbatim byte-path journaling | Write the primary's encoded journal bytes directly on both primary and replica, skipping the decode/re-encode step on the replica side and an encode-from-struct pass on the primary. Saves CPU and reduces journal-stage work on the replay-critical path. Attempted once as a replica-only optimization (branch `replica-verbatim-journal`) but created an asymmetry that diverged fsync batching behavior between primary and replica — reverted in favor of a symmetric decode/re-encode on both sides. Revisit only with a plan to apply it on both sides simultaneously, preserving per-fsync size bounds. |
| io_uring RECVSEND_FIXED_BUF | `IORING_RECVSEND_FIXED_BUF` for registered buffer recv/send. Returned EINVAL on kernel 6.8; should work on 6.10+. Also test `IORING_RECVSEND_BUNDLE` for batched recv. Est. 15-25% throughput. |
| Dual-NVMe journal hedging | Two journal threads on separate NVMe drives, response stage gates on the fastest. Cuts tail latency from P(slow) to P(slow)². Free durability redundancy. Low complexity but requires a second NVMe slot. Revisit when journal fsync is the dominant tail contributor. |
| AF_XDP transport | DaMoN '25 found AF_XDP disappoints vs DPDK for small-message request-response workloads. DPDK transport already in progress. Revisit if DPDK proves insufficient. |
| Per-account trading permissions | Gateway concern — each firm's gateway instance restricts which accounts that connection can trade. Multi-tenant access control. |
| Replica analytics (6 items) | External service — throughput counters, latency histograms, volume/book depth analytics, audit trail queries, fee/PnL. Consumes the journal stream, not engine code. |
| Output event log | Regulatory audit trail. Depends on output event channel. |
| Subscription management | Gateway concern — the engine broadcasts, the gateway filters per-subscriber. |
| Iceberg orders | Niche — only matters for venues with institutional flow. |
| Auction mechanisms | Regulated venues only. Massive complexity (state machine, indicative pricing, uncrossing). |
| Failover detection + promotion | Leader election, split-brain prevention. Distributed systems hard mode — manual promotion covers the MVP. |
| Network partition handling | Fencing, quorum-based decisions. Same as above — extremely complex. |
| Chain replication | Chain replication (primary → replica A → replica B) reduces primary fan-out. Dual parallel replication (up to 2 replicas) is implemented; chain topology is deferred. |
| Position/exposure limits | Important for derivatives, less so for spot. Defer until a derivatives buyer needs it. |
| Tiered fee schedules | Volume-based tiers and per-account overrides. Can be implemented outside Melin — a fee service looks up the account's tier and sets the rate via the existing per-instrument fee API. |
| TLS | Most exchange deployments use VLAN instead. Only needed for compliance-driven buyers. |
| Hybrid UDP multicast + TCP recovery for event channel | Current event channel is pure TCP. Multicast would reduce latency for co-located subscribers but adds complexity (gap detection, retransmit). Defer until a buyer needs sub-microsecond market data. |
