# Roadmap

Planned features sorted by value/complexity ratio for commercial readiness (exchange operators and investors).

## Active

| # | Feature | Commercial value | Complexity | Value/effort | Why |
|---|---------|:---:|:---:|:---:|-----|
| 1 | DPDK replication e2e testing | High | Low | ★★★★☆ | Test DPDK replication (smoltcp sender + receiver) on real multi-queue NICs with the bench suite. Virtual devices (TAP, af_packet) only support 1 queue so can't smoke-test locally. Branch: `feat/dpdk-bench-suite` has the implementation + bench suite integration. Needs SR-IOV hardware to validate. |
| 2 | Runtime journal rotation | High | Medium | ★★★★☆ | Rotation currently only fires at startup — a single long-running process grows the journal without bound. For a 24/5 or 24/7 exchange this is a production gap. Needs: size-threshold trigger, brief pipeline quiesce, snapshot + archive of old journal, atomic swap to a fresh file, coordination with the replica so both rotate at compatible sequence boundaries (or let each side rotate independently since their journals are byte-symmetric). Applies to both primary and replica. |
| 3 | Brand setup (domain, GitHub org, email) | Medium | Low | ★★★☆☆ | Register melin.io/melin.com, set up contact@ email, create GitHub org, transfer repo, switch commit email going forward. Do not rewrite history. |
| 4 | Safe-promotion handshake | Medium | Low | ★★★★☆ | The PROMOTE endpoint accepts the command unconditionally; the operator is responsible for verifying catch-up beforehand. The natural "wait for primary lag==0, then PROMOTE" recipe is unsafe — primary lag excludes disconnected slots from `min(slot0, slot1)`, so it can read 0 from one healthy replica alone before a freshly-spawned replacement has even connected. Make PROMOTE carry a target sequence (the last sequence the operator observed on the dying primary) and have the replica refuse promotion until its local `journal_seq` reaches it. Eliminates the silent data-loss footgun documented in `wait_for_replacement_catchup` in the failover tests. |
| 5 | Async primary ack (PLP path) | Medium | Low | ★★★★☆ | Journal writes are plain `pwrite` calls — data lands in PLP-protected drive DRAM in ~1–5µs with no OS page cache involved. The same durability argument behind `--async-replica-ack` applies symmetrically to the primary: on any single-node failure the surviving node has the data, so the primary does not need to wait for `pwrite` to return before acking clients. Adding `--async-primary-ack` would queue the journal write and immediately ack, reducing end-to-end ack latency from `pwrite (~1–5µs) + network RTT` to network RTT only when combined with `--async-replica-ack`. |

## Rumcast Hardening

Follow-ups to take the rumcast transport (Ed25519 + X25519 auth, per-session BLAKE3 keyed-MAC envelopes, multi-client demux) from feature-complete to production-ready. The foundation is on `main`; these items make it deployable for real exchange operators.

| # | Feature | Commercial value | Complexity | Value/effort | Why |
|---|---------|:---:|:---:|:---:|-----|
| 1 | Per-deployment tunables (`MAX_SESSIONS`, `IDLE_TIMEOUT`, `TERM_LENGTH`) | Medium | Low | ★★★★☆ | All three are hardcoded `const`s in `rumcast_transport.rs` (with `TODO(config)` markers from Phase 3). Operators with small boxes need lower `MAX_SESSIONS`; trading deployments with clients quietly tracking market state during off-hours need longer `IDLE_TIMEOUT`; fully-saturating 10GbE workloads need larger `TERM_LENGTH` (current 1 MiB is borderline at 1.25 MiB BDP). Mechanical refactor: promote each to `ServerConfig`, plumb through, document defaults. |
| 2 | Per-session observability via health endpoint | Medium | Low | ★★★★☆ | The rumcast `Counters` struct already aggregates per-tick stats (fragments accepted, NAKs sent, sessions created/rejected, gaps detected, etc.) and the muxers wire it through, but the existing health endpoint doesn't expose them. Operators debugging a multi-client deployment can't see per-session NAK rates, idle eviction counts, or handshake failures without reading server logs. Surface via the existing Prometheus-text health endpoint. |
| 3 | Token rotation / TTL | Medium | Medium | ★★★☆☆ | Each authenticated rumcast session derives a BLAKE3 keyed-MAC token at handshake time and uses it forever. A long-running session keeps the same key indefinitely; if the token is ever leaked (memory dump, side-channel) an attacker can forge messages until the session ends. Add server-initiated periodic re-handshake (every N min): server emits a new control message, client responds with a fresh `ChallengeResponse`, both sides switch to the new token at a fixed sequence boundary. No wire-format change to Data frames. Defense-in-depth — protects long-lived connections. |
| 4 | Operator-facing rumcast docs | High | Low | ★★★★★ | `docs/replication.md`, `docs/operations.md`, `docs/benchmarking.md`, `docs/wire-protocol.md`, `docs/security-audit.md`, and `README.md` all describe only TCP/DPDK; only `docs/roadmap.md` mentions rumcast. Operators reading the project have no doc trail showing UDP is a supported transport, what ports to firewall, what threat model applies, or how to run the bench-suite's `udp` / `udp-repl` / `udp-dual-repl` modes. Mechanical writing pass — the code and bench-suite are production-capable, the docs just haven't caught up. |
| 5 | Async / yielding catch-up on the rumcast sender | Medium | Medium | ★★★☆☆ | `replication/rumcast_sender.rs::run_catchup_or_snapshot` runs synchronously inside the single-thread tick loop. A fresh replica catching up from a 10 GiB journal is ~160k publish iterations all blocking the same thread that drains the live replica's NAKs/heartbeats. Acceptable when the other slot is empty; potentially bad when one replica is in steady-state and another joins. Fix: yield muxer ticks every N publish iterations (smaller change), or run catch-up on a per-slot worker thread with a bounded queue. |
| 6 | Promotion handoff integration test | High | Low | ★★★★☆ | `run_rumcast_replica` → `run_rumcast_primary_with_state` is wired but untested. Same for the post-`catch_unwind` promote listener path in `promote.rs`. An integration test that triggers `promote_flag` and verifies the replica binds the order-entry port + accepts a trader connection would catch regressions on a critical-path failover transition. |
| 7 | Replication-over-rumcast test breadth | Medium | Medium | ★★★☆☆ | `crates/server/tests/rumcast_replication_smoke.rs` only verifies handshake + genesis byte-equality. Untested: live data flow (orders → primary → replica DataBatch → ack chain), snapshot transfer end-to-end, replica reconnect with backoff, concurrent 2-replica setup, ring backpressure eviction, and the auth-side `Challenged` (5s) / `Authenticated` (300s) idle-timeout sweeps. The bench suite covers data flow on the LAN, but per-scenario regression tests inside `cargo test` would catch wire-format breaks earlier. |
| 8 | `SnapshotUnavailable` error frame | Low | Low | ★★★☆☆ | When the primary's `run_catchup_or_snapshot` finds `can_catch_up=false` AND no snapshot file exists, it returns false silently. The replica eventually times out + reconnects in a loop, only seeing the cause in primary-side logs. Add a `SnapshotUnavailable` `PrimaryMessage` variant so the replica logs the reason and operators can diagnose without SSH'ing into the primary. |
| 9 | `AuthFailed` integration test | Low | Low | ★★★☆☆ | `protocol/src/session.rs` has 23 unit tests for the envelope path and `crates/protocol/src/fuzz_tests.rs` is fuzzed, but there's no end-to-end test for "server rejects bad signature → client receives `AuthFailed` → connection closes cleanly". Wires the sad path. |
| 10 | Bench-suite restore-step failure handling | Low | Low | ★★★★☆ | `scripts/lan-bench-suite.sh` line ~514 swallows failures from the post-rumcast default-features rebuild via `wait "$pid" \|\| true`. If the restore fails after the rumcast build, downstream `tcp` transports in the same matrix will silently run with a rumcast-only binary. Add an explicit failure path with logging and a non-zero exit. Five-line shell fix; high value/effort because the silent-failure mode is genuinely confusing. |
| 11 | Rumcast crate stress + edge-case tests | Low | Medium | ★★☆☆☆ | The `melin-rumcast` crate has solid coverage of the happy path (wire round-trip, term rotation, NAK retransmit, flow control, multi-session demux), but three corners are untested: multicast fan-out with multi-subscriber NAK suppression + jitter convergence, term-wraparound stress (`term_id` wrap at `u32::MAX`), and the `published_fragment` HWM-based gap detection's documented "HWM doesn't decrease on interior fill" simplification. Add stress tests; add a roadmap-style code comment documenting that proper Aeron-style per-range loss detection is deferred. |
| 12 | Pool reassembly buffers in `rumcast_receiver::poll_reassembled` | Low | Low | ★★★☆☆ | Each rumcast data frame in `streaming_loop`, `wait_for_message`, and the stash path currently allocates a `Vec<u8>` (`payload.to_vec()` / `inner.to_vec()`). Fine for catch-up bursts, but on the live-streaming hot path every frame allocates. Replace with a pooled or refcounted-slice (`Bytes`-style) buffer to drop per-frame allocations to zero in steady state. Correctness is unaffected — pure throughput/latency optimization. |
| 13 | Couple rumcast keepalive interval and `LIVE_ACK_TIMEOUT` via shared constant | Low | Low | ★★★★☆ | `rumcast_receiver.rs::streaming_loop` sends keepalive acks every 1s; `rumcast_sender.rs::LIVE_ACK_TIMEOUT` evicts a slot after 3s without an ack. The 3× margin is correct but the two constants live in different files with no comment linking them — silent drift could evict healthy replicas. Promote to a shared constant in `replication/mod.rs` (or a doc-comment cross-reference) so the relationship is explicit. |
| 14 | MuxedSender active-sessions tracking | Medium | Medium | ★★★☆☆ | `MuxedSender::tick` walks every session's `drain_data` + `maybe_send_periodic` every tick, regardless of whether the session has new bytes to emit. At ~4 replicas this is invisible; at 10k+ authenticated OE-gateway sessions the per-tick HashMap walk burns L1 / branch-predictor budget on idle sessions. Maintain an `active_sessions: VecDeque<u32>` of session_ids that have published since the last drain, repopulated by `try_claim` succeeding via a back-channel SPSC or a per-session "dirty" bit. Heartbeats stay on a separate slow timer so idle sessions still keep their liveness signal. Scale-dependent — only matters once OE-gateway concurrent session counts cross ~1k. |
| 15 | Replication-sender thread split | Medium | High | ★★★☆☆ | `rumcast_sender.rs::run_sender_rumcast` runs six unrelated state machines on one thread (muxer ticks, new-session detection, inbound auth, per-slot live streaming, heartbeats, sweeps). Fine at Tier-2 throughput; at sustained 1M+ ops/s this thread becomes the bottleneck even after the sendmmsg / recv_batch wins on `main`. Split into a poller (already lives in the io_uring endpoint), an auth/control thread, and a per-slot replication-stream thread, with SPSC rings between stages. Same shape as the engine pipeline, just applied to replication. Defer until profiling shows the single-thread limit is the binding constraint. |

## Rumcast Throughput — Path to Kernel-TCP Parity

LAN bench results on the perf branch (busy-spin default, 16-client × 128-window throughput) show kernel-UDP rumcast at ~53k orders/s vs kernel-TCP (io_uring + 4 reader threads) at ~3.57M orders/s — a ~67× throughput gap and ~1.5× single-message p50 gap. The single-message gap is mostly the cost of the rumcast reliability layer (envelope + framing + flow control) and is largely structural; the throughput gap is architectural and addressable. These items, in order, close it. DPDK is sequenced last on purpose — without a known kernel-UDP per-core ceiling, DPDK numbers conflate two unknowns and become hard to defend in a sales conversation.

| # | Feature | Commercial value | Complexity | Value/effort | Why |
|---|---------|:---:|:---:|:---:|-----|
| 1 | SO_REUSEPORT session sharding (server) | High | Medium | ★★★★☆ | Today the server runs one session-translator thread that does recv → demux → app → mux → send for *all* clients — a single-core ceiling regardless of how many cores the box has. Bind N sockets to the same UDP port with `SO_REUSEPORT`; the kernel hashes flows across them. Run N session-translator shards (one per socket, pinned to its own core), each owning a disjoint set of sessions. This is the kernel-UDP equivalent of "scale out across cores" and a prerequisite for any honest comparison with multi-threaded TCP. Real customers run kernel UDP (Tier-2 commercial strategy), so this has direct revenue value. |
| 2 | Multi-threaded rumcast bench | Medium | Low | ★★★☆☆ | The bench is a single hot loop driving 16 sessions on one thread. Once #1 lands, the bench cannot saturate a sharded server without itself being multi-threaded. Drive disjoint session subsets from N bench threads with their own UDP sockets, aggregate latency histograms at the end. Without this, the server-side win in #1 is unmeasurable. |
| 3 | Wire io_uring into the rumcast hot path | Medium | Medium | ★★★☆☆ | An io_uring endpoint already exists in `crates/rumcast/src/io_uring_endpoint.rs` but isn't wired into the bench / server send/recv path the way the TCP transport's io_uring path is. Submission/completion + multi-shot recv + registered buffers eliminate per-packet `sendmmsg`/`recvmmsg` syscall cost and bring the rumcast path onto the same kernel mechanism the TCP path uses — making the kernel-UDP-vs-kernel-TCP comparison apples-to-apples. |
| 4 | DPDK kernel bypass for rumcast | High | Very high | ★★★☆☆ | Once #1–#3 ship and the kernel-UDP per-core ceiling is known, run rumcast over DPDK. Eliminates the kernel network stack entirely (no syscalls, no skb allocation, no NAPI scheduling) — leaves a clean measurement of "rumcast protocol overhead vs TCP protocol overhead" without kernel-stack noise on the UDP side. Doubles as Tier-1 acquisition optionality. Sequence after kernel-UDP optimizations so the win is attributable. |

## Deferred

Features targeting regulated venues, gateway responsibilities, or with limited near-term value. Will revisit when the core product is mature or a specific buyer requires them.

| Feature | Why deferred |
|---------|-------------|
| SPDK journal | Userspace NVMe driver for journal writes. Bypasses kernel block layer entirely. TCP pipelining already hides fsync latency (fsync and no-persist throughput converged at ~8M/s), so the kernel block layer is no longer a bottleneck. High complexity, minimal expected gain. |
| Adaptive overlapped io_uring journal writes | The non-aggressive form is **already shipped** in `run_uring`: the journal stage submits `WRITE+RWF_DSYNC` async via io_uring and continues encoding the next batch while the NVMe write is in flight. CQEs are reaped non-blocking at two points per loop iteration; backpressure only kicks in when the next batch fills before the previous fsync completes. What is still deferred is the *aggressive* double-buffer variant (always keep two writes inflight, even for tiny batches). That was built and reverted because events caught in a small overlapped batch had their cursor delayed by one extra NVMe latency, hurting p99.9. The next attempt should only overlap when `pending > 16` events. Branch preserved. |
| WRITE_FIXED for journal | Register batch buffers via `IORING_REGISTER_BUFFERS`, use `IORING_OP_WRITE_FIXED`. Skips `get_user_pages()` per SQE (~100-200ns). Failed for socket I/O on kernel 6.8 but works for file writes. |
| Vectored journal writes | Encode each event into its own stack buffer, pass as iovec array to `pwritev2`. Eliminates memcpy-per-event into the batch buffer. Small gain (~1-2µs/batch). |
| Verbatim byte-path journaling | Write the primary's encoded journal bytes directly on both primary and replica, skipping the decode/re-encode step on the replica side and an encode-from-struct pass on the primary. Saves CPU and reduces journal-stage work on the replay-critical path. Attempted once as a replica-only optimization (branch `replica-verbatim-journal`) but created an asymmetry that diverged fsync batching behavior between primary and replica — reverted in favor of a symmetric decode/re-encode on both sides. Revisit only with a plan to apply it on both sides simultaneously, preserving per-fsync size bounds. |
| io_uring RECVSEND_FIXED_BUF | `IORING_RECVSEND_FIXED_BUF` for registered buffer recv/send. Returned EINVAL on kernel 6.8; should work on 6.10+. Also test `IORING_RECVSEND_BUNDLE` for batched recv. Est. 15-25% throughput. |
| io_uring SQPOLL | `IORING_SETUP_SQPOLL` eliminates `io_uring_enter()` syscall (~1-2µs) per submission. Measured 15% p50 improvement on loopback but tail regresses on SMT-enabled machines due to SQPOLL kernel threads contending with pipeline threads. Needs Cherry server testing with SMT off and `setup_sqpoll_cpu()` pinning. Branch: `feat/uring-sqpoll`. |
| Dual-NVMe journal hedging | Two journal threads on separate NVMe drives, response stage gates on the fastest. Cuts tail latency from P(slow) to P(slow)². Free durability redundancy. Low complexity but requires a second NVMe slot. Revisit when journal fsync is the dominant tail contributor. |
| AF_XDP transport | DaMoN '25 found AF_XDP disappoints vs DPDK for small-message request-response workloads. DPDK transport already in progress. Revisit if DPDK proves insufficient. |
| Per-account trading permissions | Gateway concern — each firm's gateway instance restricts which accounts that connection can trade. Multi-tenant access control. |
| Replica analytics (6 items) | External service — throughput counters, latency histograms, volume/book depth analytics, audit trail queries, fee/PnL. Consumes the journal stream, not engine code. |
| Output event log | Regulatory audit trail. Depends on output event channel. |
| Subscription management | Gateway concern — the engine broadcasts, the gateway filters per-subscriber. |
| Iceberg orders | Niche — only matters for venues with institutional flow. |
| Auction mechanisms | Regulated venues only. Massive complexity (state machine, indicative pricing, uncrossing). |
| Replication chain hash verification | The wire protocol carries chain hashes in handshakes, heartbeats, and data batches, but the replica never verifies them against its local journal state. The primary unconditionally sends `StreamStart` without checking the replica's reported hash. Implementing verification would detect silent replica divergence (bit-rot, misrouted journal files) and trigger automatic snapshot re-sync. Deferred because validating a hash at an arbitrary historical sequence requires either a sequence→hash index or a journal re-scan. |
| Failover detection + promotion | Leader election, split-brain prevention. Distributed systems hard mode — manual promotion covers the MVP. |
| Network partition handling | Fencing, quorum-based decisions. Same as above — extremely complex. |
| Chain replication | Chain replication (primary → replica A → replica B) reduces primary fan-out. Dual parallel replication (up to 2 replicas) is implemented; chain topology is deferred. |
| Position/exposure limits | Important for derivatives, less so for spot. Defer until a derivatives buyer needs it. |
| Tiered fee schedules | Volume-based tiers and per-account overrides. Can be implemented outside Melin — a fee service looks up the account's tier and sets the rate via the existing per-instrument fee API. |
| TLS | Most exchange deployments use VLAN instead. Only needed for compliance-driven buyers. |
| Hybrid UDP multicast + TCP recovery for event channel | Current event channel is pure TCP. Multicast would reduce latency for co-located subscribers but adds complexity (gap detection, retransmit). Defer until a buyer needs sub-microsecond market data. |
