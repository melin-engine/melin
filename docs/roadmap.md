# Roadmap

Planned features sorted by value/complexity ratio for commercial readiness (exchange operators and investors).

## Active

Sequenced by dependency and priority — items higher up are unblocked
and ship first, items further down build on what came before.

| Feature | Complexity | Why |
|---------|:---:|-----|
| Publish DPDK numbers alongside kernel-TCP | Low | The README and current sales numbers are kernel-TCP only. Tier-1 TradFi venues all run on kernel bypass or specialized NICs — without DPDK figures the comparison is structurally incomplete and a buyer will discount the numbers accordingly. The DPDK transport already exists (server `dpdk` feature, `crates/exchange/bench/src/dpdk.rs`, exercised by `scripts/lan-bench-suite.sh`); this is harvesting numbers under the existing realistic flow generator and publishing them next to the kernel-TCP table in the README and `docs/benchmarking.md`. Pairs with the tick-to-trade decomposition below — without per-stage decomposition the DPDK win is reported as a black-box delta rather than attributable to NIC bypass. Independent of all engine work below. |
| Tick-to-trade decomposition in bench output | Medium | Today the bench reports full client-to-client round-trip latency. Exchange RFPs typically ask for the latency profile decomposed into NIC ingress → parse/decode → match → encode/respond → NIC egress, separately from the durability gate (journal + replication). The plumbing is partly there — `InputSlot::timestamp_ns` is captured at receipt and pipeline-stats tracks per-stage occupancy in aggregate — but per-request stage timestamps need to flow through the response path and into the bench histograms. Lets us report "gateway-to-match" the way CME / Nasdaq do, makes the kernel-TCP-vs-DPDK comparison from #2 attributable, and answers the most common diligence question without hand-waving. |
| Replicas pull configuration from primary | Medium | Today each node reads its own `ServerConfig` (instruments, fee schedules, auth keys, etc.) from local files; operators must keep the primary's and every replica's configs identical or replication diverges silently. Have the replica fetch the canonical config from the primary as part of the handshake, after auth but before journal catch-up — the replica only owns transport-level config (listen addr, replication addr, certs). Eliminates a silent-divergence class of incident, makes adding a replica a one-step operation, and is a prerequisite for the control-plane Raft item below, where the cluster owns shared state. |
| Primary-driven rotation + cross-node chain validation | Medium | A replica connecting with a divergent journal (e.g. previously synced to a different primary, corrupted history, or an ex-primary rejoining after failover with a journaled-but-unreplicated suffix) is currently accepted without warning — the primary unconditionally sends `StreamStart` and streams new events on top of stale state. The `Handshake` already carries the replica's `chain_hash` at its `last_sequence` and the `HashMismatch` frame exists in the wire protocol, but neither side acts on a mismatch. Correctness gap today, and a hard prerequisite for auto-promotion below — after a failover, the ex-primary rejoining with divergent history is the *normal* path, not an anomaly. Each node already verifies its own journal locally on every recovery (per-segment chains, cross-segment header anchors, snapshot cross-checks); what's missing is the *cross-node* comparison, and since the chain redesign made chain values segment-scoped pure functions of `(header anchor, entry bytes)`, comparing them across nodes requires shared segment boundaries. Two parts: **(a) primary-driven rotation** — replicas rotate at the sequence boundary the primary announces (a small `Rotate { boundary_seq, tail_hash }` stream frame) instead of on local triggers; a healthy replica's journal then becomes a **bitwise mirror** of the primary's, verifiable offline with a plain byte-range hash, and adopting `tail_hash` doubles as a divergence check at every rotation. **(b) Handshake + live-stream validation** — the primary recomputes its chain at the replica's `last_sequence` (containing segment's anchor + raw byte absorption, no decoding) and compares; periodic per-batch `{seq, chain_hash}` frames extend coverage to the live stream. Repair: on mismatch, route the replica through the existing snapshot-resync path and **archive** the divergent journal rather than deleting it — it is audit-trail material, and under `local` durability it may hold acked orders that did not survive the failover. Divergence-point truncation (Raft-style log rollback) stays out of scope until snapshot resync proves too slow operationally. A snapshot-bootstrapped replica's segmentation aligns at its first adopted rotation; operators can force one with `ROTATE` immediately after seeding. |
| Fencing epochs in the replication protocol | Medium | Nothing stops a paused or partitioned ex-primary from accepting orders again after a replica has been promoted — the failover playbook requires the operator to stop the old process manually, and the durability gate only protects indirectly (`hybrid`/`durably-replicated` starve without replica acks; `local` doesn't). Introduce a monotonic epoch, bumped on every promotion, persisted in the journal and carried on every replication frame and handshake: a node observing a higher epoch than its own self-demotes — stops accepting client connections and acking in-flight work — and rejoins as a replica. Closes the split-brain window for the existing manual `PROMOTE` flow on its own, and is the fencing mechanism auto-promotion below assumes (the Raft term maps onto this epoch). Independent of the chain-hash item above, though divergence repair gets cleaner once frames carry the epoch. |
| Control-plane Raft (election, membership, epochs) | High | Embed Raft across the nodes for leader election, membership, and epoch allocation only — order flow stays on the existing replication path, so the rings, ack cursors, durability gate, and snapshot/catch-up machinery survive untouched and the hot path pays nothing. Full data-plane Raft (consensus over the journal itself) was evaluated and rejected: it puts a quorum round-trip plus remote fsync on every ack, collapsing the `local`/`hybrid` durability modes into mandatory quorum durability — precisely the latency dial the product sells. Because order data is replicated out-of-band, the vote rule must be extended with journal-tip recency (candidates advertise last sequence + chain hash + epoch; voters reject candidates behind their own tip) so the most-caught-up node wins — a well-precedented shape (MongoDB's Raft-derived election over an out-of-band oplog, PacificA, Viewstamped Replication). The Raft term doubles as the fencing epoch from the item above, and the existing 1+2 topology already provides the three voters a quorum needs. Builds on config-pull, chain-hash validation, and fencing epochs above. |
| Auto-promotion via leader election | Medium | Once control-plane Raft lands, replace the operator-driven `PROMOTE` admin endpoint with election: when the leader becomes unreachable, the surviving quorum elects the most-caught-up node and the order-entry / replication addresses follow leadership automatically. Eliminates the human-in-the-loop window during failover (currently several seconds, dominated by operator reaction time) and the silent-data-loss class the manual flow guards against. Needs: leadership-aware bind/unbind on the order-entry transports, client-side reconnect to follow the new leader (the bench client and FIX gateway already reconnect on transport errors — re-resolving the leader is the new piece), and a refusal rule so a node never self-promotes when it cannot prove it holds the acked tip required by the active durability mode. Fencing comes from the epochs item above. |
| Replicas refuse gateway connections with a primary-address redirect | Low | When a client (FIX gateway, bench, native TCP client) connects to a node that isn't currently the leader, the node should reject the handshake with a structured redirect carrying the current leader's order-entry address rather than silently accepting and then dropping orders, or rejecting with an opaque error. Add a `Redirect { primary_addr }` response variant and emit it from the auth path on any non-leader node; clients reconnect to the address in the frame. Closes the "ops accidentally pointed me at a replica" footgun on its own, and pairs with auto-promotion above to give clients a clean follow-the-leader story across failover (replicas know the new leader as soon as Raft elects it; the next gateway reconnect lands in the right place automatically). Independent of the Raft/auto-promotion items — works today using static knowledge of the configured primary. |
| Revisit the `AppFactory` trait | Low | Once the full generification lift (PR-G/H/I) is in and exercised end-to-end on the trading binary plus at least one non-trading prototype, audit the trait surface for simplification. Candidates: drop `empty_for_seed` if no impl actually overrides it (its only reason to exist is the trading pre-sizing hint), fold `apply_operator_policy` into `empty`/`empty_for_seed` if no replication path genuinely needs the standalone reapply, or replace the `Vec<Event>` return of `seed_events` with an iterator if any factory ends up materialising a large seed set. Cheap pass that we couldn't make confidently during the refactor because we only had one implementor — revisit after a second one exists. |
| Offline journal inspector (`melin-admin journal-info`) | Low | The cluster-wide outage recovery playbook in `docs/replication.md` ("Cluster recovery") requires determining each node's journal end sequence by booting it briefly in `--standalone` mode and reading `journal_sequence` from `/healthz`. That's a serial boot/stop loop across every surviving node, with the side risk that an inspection node accidentally accepts client traffic or replication during the window. Add a read-only `journal-info` subcommand to the existing `melin-admin` binary (`crates/exchange/admin/src/main.rs`) that opens the journal directory directly via the existing reader (`crates/core/journal/src/reader.rs`) and prints, per node, the last sequence, the tip chain hash, and the last entry timestamp — enough to pick the longest-journal node as the new primary without starting a server process. Pairs with the recovery procedure in `docs/replication.md`; independent of all engine work above. Listed as a wishlist item in the replication doc's Limitations section — this promotes it to its own roadmap line. |
| Align `OutputSlot` to a cache line | Low | Sibling to the `InputSlot` cache-line alignment landed in `perf/input-slot-cache-line-padding`. `OutputSlot` is currently 424 B (≈6.6 cache lines), so adjacent slots in the matching→response ring straddle line boundaries and false-share across the producer (matching stage) / consumer (response stage) pair. Add `#[repr(align(64))]` on the struct and update the size assertion in `crates/exchange/server/src/exchange_app.rs` (likely 424 → 448). Layout-only change; the compile-time assertion is the test. Microbench against `perf/disruptor-slot-false-sharing-bench` first to confirm the win is comparable to the InputSlot ~16% on EPYC cross-CCD, then publish end-to-end p99 numbers. Independent of all other roadmap work. |
| Type the replica cursors in the durability gate (`WireSeq` through `ReplicationMetrics`) | Low | The gate's local `persisted` cursor is `WireSeq`-typed end-to-end, but `evaluate_durability` still flattens it into a raw `[[u64; 2]; 3]` cursor view alongside the untyped `ReplicationMetrics` gauges (`acked_sequence` / `in_memory_sequence` — the `Ack` frame's wire-seq fields, stored in the same group as the slot-acked quorum cursors). A wrong-space value wired into the view compiles silently today. Add `WireSeq`-typed accessors on `ReplicationMetrics` and carry the type through `connected_persisted_min` and the policy `CursorView` boundary. Completes the typing the `PipelineCursors` work deliberately stopped short of at the gate's multi-cursor view. |
| Unify sequence allocation (delete the wire-seq shadow) | Medium | The matching stage shadows the journal stage's sequence allocator instead of sharing one allocation point — the per-event skip rule is mirrored in two places with comments promising lockstep. v14 removed two of the three divergence sources (Checkpoint, GenesisHash) and five tests now pin the agreement (rule-table, rotation, recovery, replica-side, and the frozen-replica gate-stall integration test), so this is no longer an active bug factory — but the residual risk is concrete: a future `JournalEvent` variant with asymmetric allocate/skip rules drifts silently, because no test can feed an event kind that doesn't exist yet. Replace the shadow with a single allocation point (candidates: derive the sequence from the disruptor slot index, or allocate at publish) so the lockstep contract is deleted rather than tested. Hot-path design change — treat as a deliberate redesign, not opportunistic cleanup. **Trigger: must land before (or together with) the next `JournalEvent` variant.** Builds on the `PipelineCursors` cursor-space naming, which has landed. |
| Lockstep-confession audit | Low | The v14 bug factory had a recognizable signature: derived state maintained in parallel at a distance, with comments promising manual lockstep ("mirrors X exactly — any drift would re-introduce…"). A grep for `mirrors\|must match\|kept in sync\|stays aligned\|lockstep\|same rule as` across `crates/` finds **124 non-test hits** beyond the sites covered by the three items above — and those three came from one session's incidental reading, not a survey. Triage each hit into: benign (match-exhaustiveness notes), already pinned by a test, or v14-shaped (separately-updated state across threads/crates whose agreement nothing asserts). Output is a prioritized backlog feeding future roadmap items, plus tests or owning-module refactors for the third bucket. Cheap, bounded, and the highest-leverage way to find the next vacuous-gate-class bug before a customer does. Independent of all other roadmap work. |
| Investigate bench open-loop pacer × dpdk-dual-repl stall | Medium | The bench client's open-loop pacer (`TARGET_RATE > 0` in `crates/exchange/bench/src/dpdk.rs`) collapses to near-zero throughput specifically on the dpdk-dual-repl transport. Two reproductions on EPYC 9255: one run sustained ~14K ops/s aggregate (Outcomes 985K over 70 s wall, almost entirely during warmup), a subsequent run on a fresh reboot was 100× worse — Outcomes 9 218 over 70 s = ~130 ops/s aggregate. Closed-loop on the same setup did 2.88M ops/s ten minutes earlier, so the pipeline is healthy; the failure is in the bench client's open-loop path interacting with dpdk-dual-repl's higher per-response latency. `scheduled` and `done` counters both stay at 0; `health-poller: health scrape failed` appears in the second run, suggesting the bench host also can't reach the server's kernel-TCP health endpoint during the stall (possibly DPDK NIC capture × kernel TCP coexistence). dpdk standalone × TARGET_RATE=1M works fine on the same boxes, isolating this to the dpdk-dual-repl combo. Diagnostic shape: add periodic `info!` traces in `run_dpdk_poll_bench` for per-conn `inflight.len`, `pop_due_some/none` counts, and send/recv byte counts; rerun with `RUST_LOG=info`; identify where the throughput drops out (pacer never firing? send returning Ok(0)? recv starvation?). Independent of all engine work above. Doesn't affect closed-loop benches, which we use throughout. |
| Document the in-flight order contract on pipeline halt | Low | The persist-before-ack gate (`crates/core/transport-core/src/pipeline.rs:746-750`) guarantees no client is acked for an event that isn't durable on disk (and on a replica's disk, per the active durability mode). What is **not** documented is the symmetric contract for orders that were submitted but never acked: when the journal stage exits on disk failure, the response stage cannot advance past the failed batch and in-flight responses never go out — clients see only a TCP RST when the process exits. There is no structured `JournalUnavailable` reject. The reconciliation contract is currently implicit: an order whose connection dropped mid-flight may have been (a) not journaled — safe to retry; (b) journaled and matched but never acked — exchange state has already advanced, must not retry blindly; or (c) journaled but not yet matched — will be deterministically replayed on primary restart. Clients have to reconcile via account state on reconnect to disambiguate. Today this is inferable only from reading the pipeline source. Exchange RFPs ask for it in writing. Add a section to `docs/journal.md` (or a new `docs/client-recovery.md`) that spells out the contract with worked examples, and cross-link from `docs/replication.md`. No code change. Independent of all other roadmap work. |
| Split replication off the DPDK client poll thread | Medium | On the DPDK path the replication driver runs **inline on the client poll loop**: `DpdkReplicationDriver::tick` is invoked from the same `dpdk_transport.rs` iteration that drains client responses to the wire and pulls client order packets off the NIC (`transport.poll()`), and the replication listener deliberately shares the trading queue (`server.rs`: the poll thread "accepts both trading and replication connections off the same queue"). This couples replication and the client hot path in **both directions**. **(a) Clients starve replication:** under heavy client load the replication sender doesn't get enough cycles to drain the ring, it fills, and a healthy replica is evicted. **(b) Replication stalls clients:** a replica handshake runs journal catch-up / snapshot transfer, and the catch-up→live handoff back-fill (`bridge_catchup_to_live`, bounded by `HANDOFF_BRIDGE_TIMEOUT`), inline on this poll thread — so for that duration the thread stops ingesting orders and sending acks for the clients on its queue. The matching/journal/response stages run on their own cores and keep processing already-ingested work; it is the NIC-facing RX/TX on the shared poll thread that pauses. Bounded today — the handoff spin is capped at 30 ms (typically microseconds on PLP+xfs), catch-up is the larger but pre-existing stall, and the kernel-TCP path is unaffected (each replica gets its own handler thread) — so **not urgent**. Fix: give replication its own core — a separate smoltcp socket set on a dedicated DPDK queue, or a shared-nothing handoff from the client poll thread — preserving the single-writer invariant on the replication ring (consumer side only). Promote when DPDK dual-replica reconnect latency becomes a diligence concern. Surfaced by the `perf/handoff-deterministic-close` review; promotes the former "Split the DPDK poll thread" optimization item with the replication-blocks-clients direction added. |

| # | Feature | Commercial value | Complexity | Value/effort | Why |

| 2 | IPv6 support | Medium | Low | ★★★☆☆ | `server_addr` and `listen_addr` are IPv4-only today (validation rejects IPv6). Many modern data centers require IPv6 dual-stack. |
| 3 | Market data (35=V/W/X) | Medium | High | ★★☆☆☆ | MarketDataRequest, snapshot/full refresh, incremental refresh. Requires a feed builder that consumes the engine's output event channel and maintains per-subscription book state. Larger surface than order entry. |

## DPDK Transport Optimization

| # | Optimization | Est. impact | Complexity | Description |
|---|-------------|------------|------------|-------------|
| 1 | Bypass smoltcp on hot path | Significant latency | Very high | For connected+authenticated clients, parse TCP directly from raw Ethernet frames. Eliminates smoltcp's per-packet overhead (neighbor lookup, socket dispatch, congestion window, timer checks). Custom minimal TCP for steady-state data path only. |
| 2 | Unified transport abstraction over io_uring and DPDK | Moderate maintainability, moderate latency | High | The TCP (io_uring) and DPDK code paths have independent replication senders, response writers, and connection handlers with duplicated logic that drifts (e.g. the journal-recreate-on-reconnect bug was fixed in TCP but not DPDK). Investigate a transport trait that captures the common interface (accept, connect, send, recv, poll) so the replication protocol, response stage, and connection state machine can be written once. The challenge is that io_uring is completion-based (submit SQE, reap CQE) while DPDK/smoltcp is poll-based (call poll, check buffers) — the abstraction must not penalize either model. |

## Deferred

Features targeting regulated venues, gateway responsibilities, or with limited near-term value. Will revisit when the core product is mature or a specific buyer requires them.

| Feature | Why deferred |
|---------|-------------|
| SPDK journal | Userspace NVMe driver for journal writes. Bypasses kernel block layer entirely. TCP pipelining already hides fsync latency (fsync and no-persist throughput converged at ~8M/s), so the kernel block layer is no longer a bottleneck. High complexity, minimal expected gain. |
| Adaptive overlapped io_uring journal writes | The non-aggressive form is **already shipped** in `run_uring`: the journal stage submits `WRITE+RWF_DSYNC` async via io_uring and continues encoding the next batch while the NVMe write is in flight. CQEs are reaped non-blocking at two points per loop iteration; backpressure only kicks in when the next batch fills before the previous fsync completes. What is still deferred is the *aggressive* double-buffer variant (always keep two writes inflight, even for tiny batches). That was built and reverted because events caught in a small overlapped batch had their cursor delayed by one extra NVMe latency, hurting p99.9. The next attempt should only overlap when `pending > 16` events. Branch preserved. |
| WRITE_FIXED for journal | Register batch buffers via `IORING_REGISTER_BUFFERS`, use `IORING_OP_WRITE_FIXED`. Skips `get_user_pages()` per SQE (~100-200ns). Failed for socket I/O on kernel 6.8 but works for file writes. |
| Vectored journal writes | Encode each event into its own stack buffer, pass as iovec array to `pwritev2`. Eliminates memcpy-per-event into the batch buffer. Small gain (~1-2µs/batch). |
| Verbatim byte-path journaling | Write the primary's encoded journal bytes directly on both primary and replica, skipping the decode/re-encode step on the replica side and an encode-from-struct pass on the primary. Saves CPU and reduces journal-stage work on the replay-critical path. Attempted once as a replica-only optimization (branch `replica-verbatim-journal`) but created an asymmetry that diverged fsync batching behavior between primary and replica — reverted in favor of a symmetric decode/re-encode on both sides. Revisit only with a plan to apply it on both sides simultaneously, preserving per-fsync size bounds. |
| Shared journal-writer core (`SectorWriter` / `BufferedWriter`) | The two writers duplicate the framing/sequence/replication-slice state machine and genuinely differ only in I/O strategy (O_DIRECT sector management vs buffered). The actively drift-prone part is already gone — v14 deleted the shared checkpoint policy and extracted the chain into `chain.rs`, and `pipeline_journal_contents_match_across_writer_modes` pins cross-writer equivalence — so what remains is duplication tax, not a bug factory. v14 paid that tax visibly (481 + 615 lines changed in parallel across the two writers). Deferred with a rule rather than a date: the next journal feature that requires editing both writers does the extraction *first*, then implements the feature once. Doing it cold on the hot path buys little and risks more than it saves. |
| io_uring RECVSEND_FIXED_BUF | `IORING_RECVSEND_FIXED_BUF` for registered buffer recv/send. Returned EINVAL on kernel 6.8; should work on 6.10+. Also test `IORING_RECVSEND_BUNDLE` for batched recv. Est. 15-25% throughput. |
| Dual-NVMe journal hedging | Two journal threads on separate NVMe drives, response stage gates on the fastest. Cuts tail latency from P(slow) to P(slow)². Free durability redundancy. Low complexity but requires a second NVMe slot. Revisit when journal fsync is the dominant tail contributor. |
| AF_XDP transport | DaMoN '25 found AF_XDP disappoints vs DPDK for small-message request-response workloads. DPDK transport already in progress. Revisit if DPDK proves insufficient. |
| Per-account trading permissions | Gateway concern — each firm's gateway instance restricts which accounts that connection can trade. Multi-tenant access control. |
| Replica analytics (6 items) | External service — throughput counters, latency histograms, volume/book depth analytics, audit trail queries, fee/PnL. Consumes the journal stream, not engine code. |
| Output event log | Regulatory audit trail. Depends on output event channel. |
| Subscription management | Gateway concern — the engine broadcasts, the gateway filters per-subscriber. |
| Iceberg orders | Niche — only matters for venues with institutional flow. |
| Auction mechanisms | Regulated venues only. Massive complexity (state machine, indicative pricing, uncrossing). |
| Chain replication | Chain replication (primary → replica A → replica B) reduces primary fan-out. Dual parallel replication (up to 2 replicas) is implemented; chain topology is deferred. |
| Position/exposure limits | Important for derivatives, less so for spot. Defer until a derivatives buyer needs it. |
| Tiered fee schedules | Volume-based tiers and per-account overrides. Can be implemented outside Melin — a fee service looks up the account's tier and sets the rate via the existing per-instrument fee API. |
| TLS | Most exchange deployments use VLAN instead. Only needed for compliance-driven buyers. |
| Hybrid UDP multicast + TCP recovery for event channel | Current event channel is pure TCP. Multicast would reduce latency for co-located subscribers but adds complexity (gap detection, retransmit). Defer until a buyer needs sub-microsecond market data. |
