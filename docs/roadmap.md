# Roadmap

Planned features sorted by value/complexity ratio for commercial readiness (exchange operators and investors).

## Active

Sequenced by dependency and priority — items higher up are unblocked
and ship first, items further down build on what came before.

| Feature | Complexity | Why |
|---------|:---:|-----|
| Publish DPDK numbers alongside kernel-TCP | Low | The README and current sales numbers are kernel-TCP only. Tier-1 TradFi venues all run on kernel bypass or specialized NICs — without DPDK figures the comparison is structurally incomplete and a buyer will discount the numbers accordingly. The DPDK transport already exists (server `dpdk` feature, `crates/exchange/bench/src/dpdk.rs`, exercised by `scripts/lan-bench-suite.sh`); this is harvesting numbers under the existing realistic flow generator and publishing them next to the kernel-TCP table in the README and `docs/benchmarking.md`. Pairs with #3 — without per-stage decomposition the DPDK win is reported as a black-box delta rather than attributable to NIC bypass. Independent of all engine work below. |
| Tick-to-trade decomposition in bench output | Medium | Today the bench reports full client-to-client round-trip latency. Exchange RFPs typically ask for the latency profile decomposed into NIC ingress → parse/decode → match → encode/respond → NIC egress, separately from the durability gate (journal + replication). The plumbing is partly there — `InputSlot::timestamp_ns` is captured at receipt and pipeline-stats tracks per-stage occupancy in aggregate — but per-request stage timestamps need to flow through the response path and into the bench histograms. Lets us report "gateway-to-match" the way CME / Nasdaq do, makes the kernel-TCP-vs-DPDK comparison from #2 attributable, and answers the most common diligence question without hand-waving. |
| Replicas pull configuration from primary | Medium | Today each node reads its own `ServerConfig` (instruments, fee schedules, auth keys, etc.) from local files; operators must keep the primary's and every replica's configs identical or replication diverges silently. Have the replica fetch the canonical config from the primary as part of the handshake, after auth but before journal catch-up — the replica only owns transport-level config (listen addr, replication addr, certs). Eliminates a silent-divergence class of incident, makes adding a replica a one-step operation, and is a prerequisite for any meaningful Raft integration (#8) where the cluster owns shared state. |
| Raft cluster integration | Very high | Replace the bespoke 1+1 / 1+2 replication with Raft consensus over the journal. Gives well-known durability guarantees (regulators recognise it), automatic leader election, formal split-brain prevention, and unlocks 3+ replica topologies — currently structurally impossible. Major undertaking: needs a Raft library (or in-house implementation), integration with the event-sourced pipeline so log entries map onto journal events, and a migration story for existing primary+replica deployments. Prerequisite for the auto-promotion item below. |
| Auto-promotion via Raft leader election | Medium | Once #8 (Raft) lands, replace the operator-driven `PROMOTE` admin endpoint with Raft's built-in leader-election semantics: when the current leader becomes unreachable, the surviving quorum elects a new one and the order-entry / replication addresses follow leadership automatically. Eliminates the human-in-the-loop window during failover (currently several seconds, dominated by operator reaction time) and the silent-data-loss class the manual flow guards against. Needs: leadership-aware bind/unbind on the order-entry transports, client-side reconnect to follow the new leader (the bench client and FIX gateway already reconnect on transport errors — re-resolving the leader is the new piece), and a fencing mechanism so a partitioned ex-leader stops acking before its lease expires. |
| Replicas refuse gateway connections with a primary-address redirect | Low | When a client (FIX gateway, bench, native TCP client) connects to a node that isn't currently the leader, the node should reject the handshake with a structured redirect carrying the current leader's order-entry address rather than silently accepting and then dropping orders, or rejecting with an opaque error. Add a `Redirect { primary_addr }` response variant and emit it from the auth path on any non-leader node; clients reconnect to the address in the frame. Closes the "ops accidentally pointed me at a replica" footgun on its own, and pairs with #9 to give clients a clean follow-the-leader story across failover (replicas know the new leader as soon as Raft elects it; the next gateway reconnect lands in the right place automatically). Independent of #8/#9 — works today using static knowledge of the configured primary. |
| Cleaner separation between the LMAX core and the exchange | Medium | The `crates/core/` ↔ `crates/exchange/` split is already in place at the type level — `melin-app`, `melin-pipeline`, `melin-journal`, `melin-transport-core`, `melin-dpdk` carry no exchange dependencies, and `JournalEvent<E: AppEvent>` is properly parameterised over the consumer's event type. What still leaks is the language: ~140 references to "order", "fill", "cancel", "account", "symbol", "trade" in core comments and identifiers (e.g. `crates/core/transport-core/src/pipeline.rs` describes the input batch in terms of "incoming client orders" and "fan-out reports (fills, acks, cancels)"). Audit pass: rename exchange-flavoured comment/identifier vocabulary to domain-neutral terms ("commands", "events", "reports"), tighten the `AppEvent` trait surface so any remaining domain assumptions are explicit, and document the core/exchange contract in a top-level README so a buyer evaluating just the LMAX framework can read it without wading through matching-engine prose. Commercial value: the core could be sold or licensed independently — as a deterministic event-sourced pipeline with replication and durability — to non-exchange buyers (clearinghouses, risk engines, generic LMAX-style consumers). Independent of all engine work above. |
| Generify the server orchestrator over `<A: Application>` | Medium | The runtime leaves and replication pipeline are now generic (PR-GH.1 / .2 in the recent refactor series); the orchestrator in `crates/exchange/server/src/runtime/server.rs` is the last concrete piece. Three trading-specific constructions block generification: `ExchangeResponseEncoder` and `ExchangeRequestDecoder` are instantiated inline (run_as_primary / run_dpdk_impl), and `spawn_event_publisher` calls a market-data firehose hard-wired to the trading event channel. Lift all three to `run()` / `run_dpdk()` arguments (the decoder and encoder fit as `Arc<dyn …>` like the factory; the event publisher is a thread-spawning closure `Option<Box<dyn FnOnce(Consumer<OutputSlot<A::Report, A::QueryResponse>>, …) -> JoinHandle<()> + Send + 'static>>`). Once threaded through, server.rs becomes `run<A: Application>(…)` and the runtime can move to `crates/core/server-runtime/`. Drop the trading aliases from the server crate's lib.rs at the same time (App / TradingEvent / InputSlot / OutputSlot / JournalEvent / JournalStage / SectorWriter / BufferedWriter / JournalReader) and update the external callers (bench, tests, examples) to spell out concrete trading types. |
| Revisit the `AppFactory` trait | Low | Once the full generification lift (PR-G/H/I) is in and exercised end-to-end on the trading binary plus at least one non-trading prototype, audit the trait surface for simplification. Candidates: drop `empty_for_seed` if no impl actually overrides it (its only reason to exist is the trading pre-sizing hint), fold `apply_operator_policy` into `empty`/`empty_for_seed` if no replication path genuinely needs the standalone reapply, or replace the `Vec<Event>` return of `seed_events` with an iterator if any factory ends up materialising a large seed set. Cheap pass that we couldn't make confidently during the refactor because we only had one implementor — revisit after a second one exists. |
| Brand setup (domain, GitHub org, email) | Low | Register melin.io/melin.com, set up contact@ email, create GitHub org, transfer repo, switch commit email going forward. Do not rewrite history. Independent of the engine work above; sequenced last because it doesn't unblock anything technical. |
| Re-enable snapshot/journal chain-hash cross-check on recovery | Low | The chain-hash comparison was removed because the shadow stage was saving ring-space cursors instead of journal sequences. That root cause is now fixed (FsyncState provides self-consistent snapshots). Re-adding the comparison for app-event anchors is straightforward (~10 LOC in `replay_segment`). Checkpoint-anchored snapshots would need the reader to expose its chain hash at control-entry boundaries — `next_entry` processes those internally and never surfaces them to the caller. App-event-only coverage is sufficient in practice: two independent clusters producing the same sequence at the same Checkpoint boundary with a matching BLAKE3 prefix is astronomically unlikely. |
| Replica chain-hash validation at handshake | Medium | A replica connecting with a divergent journal (e.g. previously synced to a different primary, or with corrupted history) is currently accepted without warning — the primary unconditionally sends `StreamStart` and streams new events on top of stale state. The `Handshake` already carries the replica's `chain_hash` at its `last_sequence` and the `HashMismatch` frame exists in the wire protocol, but neither side acts on a mismatch. Correctness gap: post-failover, the promoted node holds a journal that doesn't match the events clients were told about. Two implementation shapes: **(a) tip-only check** — primary sends its current `chain_hash` + tip sequence in `StreamStart`, replica verifies when its `last_sequence == primary's tip`; catches the common reconnect case (~150 LOC, half a day, no journal-crate changes); **(b) full arbitrary-N check** — new journal API that scans from the nearest Checkpoint ≤ `last_sequence` forward, reproducing the writer's BLAKE3 accumulation rule bit-for-bit; catches divergent replicas that are also behind (~320 LOC, 1–2 days, journal-crate surface area). Recommend (a) first to close the high-frequency gap, (b) when a customer needs hardened bit-rot detection across long reconnect gaps. Independent of all engine work above. |
| Investigate replica-journal divergence at segment boundaries | Medium | Running `scripts/journal-verify.sh` against any tcp-dual-repl bench shows `error at entry 1: sequence duplicate: <N> already seen (immediately after seq <N>)` on **both** replicas, while the primary's journal verifies cleanly. The pattern (two entries at the same sequence, at offset 1 of the *live* segment — i.e. the most-recently-rotated file) reproduces identically on pre- and post-prestage builds, so it predates the rotation fast path and is not caused by it. The verify script reports `PARTIAL MATCH` and exits 0 when chain hashes match, which is why no prior bench surfaced it. Hypothesis: the replica receiver's writer re-emits a `GenesisHash` (or analogous transparent entry) at the boundary sequence on rotation, then writes the next replicated event at the same seq instead of `seq+1`. Investigation needs: dump entries 1 and 2 of a replica live segment, compare to the corresponding bytes on primary, trace the receiver's per-event write path (`crates/exchange/server/src/replication/tcp_receiver.rs`) across a rotation. Correctness gap if real: a verified-bad replica could be promoted and start serving from a journal that doesn't match what primary acked. Closes one of the standing "replicas verify clean" claims in the README. |
| Align `OutputSlot` to a cache line | Low | Sibling to the `InputSlot` cache-line alignment landed in `perf/input-slot-cache-line-padding`. `OutputSlot` is currently 424 B (≈6.6 cache lines), so adjacent slots in the matching→response ring straddle line boundaries and false-share across the producer (matching stage) / consumer (response stage) pair. Add `#[repr(align(64))]` on the struct and update the size assertion in `crates/exchange/server/src/exchange_app.rs` (likely 424 → 448). Layout-only change; the compile-time assertion is the test. Microbench against `perf/disruptor-slot-false-sharing-bench` first to confirm the win is comparable to the InputSlot ~16% on EPYC cross-CCD, then publish end-to-end p99 numbers. Independent of all other roadmap work. |
| Durability-policy degraded-duration counter | Low | `melin_durability_policy_degraded` is currently a 0/1 gauge (`crates/exchange/server/src/health.rs:421`), which means SLO dashboards can only sample point-in-time state — they can't compute time-in-degraded over a window without scraping at high frequency and reconstructing intervals client-side. Pair the gauge with a `melin_durability_policy_degraded_seconds_total` counter that ticks while the policy is clamped below its target. Standard Prometheus pattern (gauge + accumulated-duration counter). Lets operators write `rate(...degraded_seconds_total[5m])` for "fraction of time degraded in the last 5 min" without ad-hoc tooling. ~30 LoC plus a test. Independent of all other roadmap work. |
| Investigate bench open-loop pacer × dpdk-dual-repl stall | Medium | The bench client's open-loop pacer (`TARGET_RATE > 0` in `crates/exchange/bench/src/dpdk.rs`) collapses to near-zero throughput specifically on the dpdk-dual-repl transport. Two reproductions on EPYC Cherry: one run sustained ~14K ops/s aggregate (Outcomes 985K over 70 s wall, almost entirely during warmup), a subsequent run on a fresh reboot was 100× worse — Outcomes 9 218 over 70 s = ~130 ops/s aggregate. Closed-loop on the same setup did 2.88M ops/s ten minutes earlier, so the pipeline is healthy; the failure is in the bench client's open-loop path interacting with dpdk-dual-repl's higher per-response latency. `scheduled` and `done` counters both stay at 0; `health-poller: health scrape failed` appears in the second run, suggesting the bench host also can't reach the server's kernel-TCP health endpoint during the stall (possibly DPDK NIC capture × kernel TCP coexistence). dpdk standalone × TARGET_RATE=1M works fine on the same boxes, isolating this to the dpdk-dual-repl combo. Diagnostic shape: add periodic `info!` traces in `run_dpdk_poll_bench` for per-conn `inflight.len`, `pop_due_some/none` counts, and send/recv byte counts; rerun with `RUST_LOG=info`; identify where the throughput drops out (pacer never firing? send returning Ok(0)? recv starvation?). Independent of all engine work above. Doesn't affect closed-loop benches, which we use throughout. |
| Document the in-flight order contract on pipeline halt | Low | The persist-before-ack gate (`crates/core/transport-core/src/pipeline.rs:746-750`) guarantees no client is acked for an event that isn't durable on disk (and on a replica's disk, per the active durability mode). What is **not** documented is the symmetric contract for orders that were submitted but never acked: when the journal stage exits on disk failure, the response stage cannot advance past the failed batch and in-flight responses never go out — clients see only a TCP RST when the process exits. There is no structured `JournalUnavailable` reject. The reconciliation contract is currently implicit: an order whose connection dropped mid-flight may have been (a) not journaled — safe to retry; (b) journaled and matched but never acked — exchange state has already advanced, must not retry blindly; or (c) journaled but not yet matched — will be deterministically replayed on primary restart. Clients have to reconcile via account state on reconnect to disambiguate. Today this is inferable only from reading the pipeline source. Exchange RFPs ask for it in writing. Add a section to `docs/journal.md` (or a new `docs/client-recovery.md`) that spells out the contract with worked examples, and cross-link from `docs/replication.md`. No code change. Independent of all other roadmap work. |

## Deferred

Features targeting regulated venues, gateway responsibilities, or with limited near-term value. Will revisit when the core product is mature or a specific buyer requires them.

| Feature | Why deferred |
|---------|-------------|
| SPDK journal | Userspace NVMe driver for journal writes. Bypasses kernel block layer entirely. TCP pipelining already hides fsync latency (fsync and no-persist throughput converged at ~8M/s), so the kernel block layer is no longer a bottleneck. High complexity, minimal expected gain. |
| Adaptive overlapped io_uring journal writes | The non-aggressive form is **already shipped** in `run_uring`: the journal stage submits `WRITE+RWF_DSYNC` async via io_uring and continues encoding the next batch while the NVMe write is in flight. CQEs are reaped non-blocking at two points per loop iteration; backpressure only kicks in when the next batch fills before the previous fsync completes. What is still deferred is the *aggressive* double-buffer variant (always keep two writes inflight, even for tiny batches). That was built and reverted because events caught in a small overlapped batch had their cursor delayed by one extra NVMe latency, hurting p99.9. The next attempt should only overlap when `pending > 16` events. Branch preserved. |
| WRITE_FIXED for journal | Register batch buffers via `IORING_REGISTER_BUFFERS`, use `IORING_OP_WRITE_FIXED`. Skips `get_user_pages()` per SQE (~100-200ns). Failed for socket I/O on kernel 6.8 but works for file writes. |
| Vectored journal writes | Encode each event into its own stack buffer, pass as iovec array to `pwritev2`. Eliminates memcpy-per-event into the batch buffer. Small gain (~1-2µs/batch). |
| Verbatim byte-path journaling | Write the primary's encoded journal bytes directly on both primary and replica, skipping the decode/re-encode step on the replica side and an encode-from-struct pass on the primary. Saves CPU and reduces journal-stage work on the replay-critical path. Attempted once as a replica-only optimization (branch `replica-verbatim-journal`) but created an asymmetry that diverged fsync batching behavior between primary and replica — reverted in favor of a symmetric decode/re-encode on both sides. Revisit only with a plan to apply it on both sides simultaneously, preserving per-fsync size bounds. |
| io_uring RECVSEND_FIXED_BUF | `IORING_RECVSEND_FIXED_BUF` for registered buffer recv/send. Returned EINVAL on kernel 6.8; should work on 6.10+. Also test `IORING_RECVSEND_BUNDLE` for batched recv. Est. 15-25% throughput. |
| Dual-NVMe journal hedging | Two journal threads on separate NVMe drives, response stage gates on the fastest. Cuts tail latency from P(slow) to P(slow)². Free durability redundancy. Low complexity but requires a second NVMe slot. Revisit when journal fsync is the dominant tail contributor. |
| AF_XDP transport | DaMoN '25 found AF_XDP disappoints vs DPDK for small-message request-response workloads. DPDK transport already in progress. Revisit if DPDK proves insufficient. |
| Per-account trading permissions | Gateway concern — each firm's gateway instance restricts which accounts that connection can trade. Multi-tenant access control. |
| Replica analytics (6 items) | External service — throughput counters, latency histograms, volume/book depth analytics, audit trail queries, fee/PnL. Consumes the journal stream, not engine code. |
| Output event log | Regulatory audit trail. Depends on output event channel. |
| Subscription management | Gateway concern — the engine broadcasts, the gateway filters per-subscriber. |
| Iceberg orders | Niche — only matters for venues with institutional flow. |
| Auction mechanisms | Regulated venues only. Massive complexity (state machine, indicative pricing, uncrossing). |
| Failover detection + promotion | Leader election, split-brain prevention. Distributed systems hard mode — manual promotion covers the MVP. |
| Network partition handling | Fencing, quorum-based decisions. Same as above — extremely complex. |
| Chain replication | Chain replication (primary → replica A → replica B) reduces primary fan-out. Dual parallel replication (up to 2 replicas) is implemented; chain topology is deferred. |
| Position/exposure limits | Important for derivatives, less so for spot. Defer until a derivatives buyer needs it. |
| Tiered fee schedules | Volume-based tiers and per-account overrides. Can be implemented outside Melin — a fee service looks up the account's tier and sets the rate via the existing per-instrument fee API. |
| TLS | Most exchange deployments use VLAN instead. Only needed for compliance-driven buyers. |
| Hybrid UDP multicast + TCP recovery for event channel | Current event channel is pure TCP. Multicast would reduce latency for co-located subscribers but adds complexity (gap detection, retransmit). Defer until a buyer needs sub-microsecond market data. |
